# 测试用例格式标准

## 1. 分层结构

| 层级 | 英文 | 中文 | 定义 | 示例 |
|------|------|------|------|------|
| L1 | Module | 模块 | 业务模块 | 登录功能、订单管理 |
| L2 | Scenario | 场景 | 独立操作路径 | 用户名密码登录、手机验证码登录 |
| L3 | Case | 用例 | 具体测试执行单位 | 验证有效用户名和密码登录成功 |

**层级关系**：
```
Module（模块）
├── Scenario（场景）
│   ├── Case（用例）
│   ├── Case（用例）
│   └── ...
└── Scenario（场景）
    └── ...
```

**划分原则**：
- Module 按业务模块划分
- Scenario 按独立操作路径划分（不是数据差异）
- Case 按等价类/边界值划分

---

## 2. 用例格式

```markdown
## [优先级] 用例标题
[测试类型] 类型值
[前置条件] 前置条件描述
[测试步骤] 1. 步骤1。2. 步骤2。3. 步骤3
[预期结果] 1. 结果1。2. 结果2。3. 结果3
```

反向用例：

```markdown
## [优先级][反向] 用例标题
[测试类型] 类型值
[前置条件] 前置条件描述
[测试步骤] 1. 步骤1。2. 步骤2。3. 步骤3
[预期结果] 1. 结果1。2. 结果2。3. 结果3
```

---

## 3. 字段说明

### 3.1 优先级（必填）

| 值 | 定义             |
| -- | ---------------- |
| P1 | 核心功能正向流程 |
| P2 | 基本功能正向流程 |
| P3 | 核心功能异常场景 |
| P4 | 边界条件         |
| P5 | 低频场景         |

### 3.2 [反向] 标记（可选）

标识反向测试用例，紧跟优先级，无空格。

### 3.3 用例标题

- 以"验证"开头
- 描述测试目的

### 3.4 测试类型（必填）

12 种枚举值：

| 类型                 |
| -------------------- |
| 功能                 |
| 兼容性               |
| 易用性               |
| 性能                 |
| 稳定性               |
| 安全性               |
| 可靠性               |
| 效果（AI类、资源类） |
| 效果（硬件器件类）   |
| 可维护性             |
| 可移植性             |
| 埋点                 |

注意：必须完全匹配，括号用中文括号。

### 3.5 前置条件（非必填）

测试执行前的准备工作。如无特殊前置条件（如默认进入某页面），可省略。若填写，必须具体。

### 3.6 测试步骤（必填）

- 编号从 1 开始，连续递增
- 中文句号分隔
- 数据必须具体
- **每个步骤到达一个验证点**，多个操作可合并为一个步骤

### 3.7 预期结果（必填）

- 编号从 1 开始，连续递增
- 中文句号分隔
- **数量必须与测试步骤一致**
- 必须可验证

## 4. 数据要求

### 4.1 测试数据尽可能具体

如果能确认具体数据，就用具体数据；无法确认时可用描述性文字。

| 正确        | 错误       |
| ----------- | ---------- |
| test_user   | {username} |
| Test1234    | {password} |
| 13800138000 | {phone}    |

### 4.2 预期结果可验证

根据实际情况决定详细程度。简单场景可简写，复杂场景需具体。

| 简写     | 具体                                   |
| -------- | -------------------------------------- |
| 登录成功 | 跳转到/home页面，显示"欢迎，test_user" |
| 提示错误 | 显示红色提示"用户名不能为空"           |

### 4.3 不确定点标记

无法确定的细节用 `[待确认]` 标记：

```markdown
[预期结果] 1. 订单创建成功。2. [待确认] 库存扣减数量
```

## 5. 目录结构

```
test-case/
├── plan.md                      # 测试规划（Module/Scenario 定义）
└── {Module}/                    # 模块目录
    ├── {Scenario}.md           # 场景文件（包含多个 Case）
    └── ...
```

**示例**：

```
test-case/
├── plan.md
├── 登录功能/
│   ├── 用户名密码登录.md
│   ├── 手机验证码登录.md
│   └── 第三方登录.md
└── 订单管理/
    ├── 创建订单.md
    └── 取消订单.md
```

**命名规则**：
- Module 目录：中文名称，与 plan.md 中一致
- Scenario 文件：中文名称 + `.md` 后缀
- 一个 Scenario 文件包含该场景下的所有 Case

## 6. 示例

### 6.1 正向用例（单验证点）

```markdown
## [P1] 验证有效用户名和密码登录成功
[测试类型] 功能
[前置条件] 已注册用户test_user，密码Test1234
[测试步骤] 1. 输入用户名test_user，输入密码Test1234，点击登录
[预期结果] 1. 登录成功，跳转到/home页面
```

### 6.2 正向用例（多验证点）

```markdown
## [P1] 验证登录后查看用户信息
[测试类型] 功能
[前置条件] 已注册用户test_user，密码Test1234
[测试步骤] 1. 输入用户名test_user，输入密码Test1234，点击登录。2. 点击用户头像
[预期结果] 1. 登录成功，跳转到/home页面。2. 显示用户信息，用户名为test_user
```

### 6.3 反向用例

```markdown
## [P3][反向] 验证用户名为空时登录失败
[测试类型] 功能
[前置条件] 进入登录页
[测试步骤] 1. 不输入用户名，输入密码Test1234，点击登录
[预期结果] 1. 登录失败，提示"用户名不能为空"
```

### 6.4 边界用例

```markdown
## [P4] 验证密码长度下边界（8位）登录成功
[测试类型] 功能
[前置条件] 已注册用户test_user，密码Abcd1234（8位）
[测试步骤] 1. 输入用户名test_user，输入密码Abcd1234，点击登录
[预期结果] 1. 登录成功，跳转到/home页面
```
