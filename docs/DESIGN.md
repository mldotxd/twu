# TWU 技术设计文档

> Test Workflow Utility - 基于上下文工程的人机协作测试流程

## 核心理念

### 上下文工程 (Context Engineering)

传统的 Prompt Engineering 关注单次对话的提示词优化，而**上下文工程**关注的是：

1. **结构化的任务定义**：通过 SKILL.md 定义每个阶段的输入、输出、流程、检查清单
2. **精确的上下文注入**：AI 在执行任务时，获得完整的领域知识和操作规范
3. **可复现的工作流**：相同的输入 + 相同的 SKILL = 一致的输出质量

```
传统方式：用户 → 模糊指令 → AI → 不确定的输出
上下文工程：用户 → 触发 Skill → SKILL.md 注入上下文 → AI → 规范化输出
```

### HITL (Human-in-the-Loop)

AI 不是替代人，而是**辅助人**。每个阶段都设计了人机协作的闭环：

```
AI 生成 → 人工审核 → AI 理解改动 → 确认闭环 → 下一阶段
```

关键设计：
- **Git 作为协作媒介**：AI 生成后 commit，人工修改后 AI 通过 `git diff` 理解改动
- **显式的审核节点**：每个 Phase 结束都有明确的人工审核环节
- **渐进式确认**：不是一次性生成全部，而是分阶段确认

## 四个阶段详解

### 1. 需求解析 (req-parse)

**目标**：将原始文档转化为标准化的需求文档

```
输入：raw/ (PDF/Word/图片/Markdown)
输出：req/index.md
```

**人机协作流程**：

| 步骤 | 执行者 | 动作 |
|------|--------|------|
| 1 | AI | 解析文档（Docling/多模态分析） |
| 2 | AI | 生成 `req/index.md`，标记 `[待确认]` |
| 3 | AI | `git commit` |
| 4 | 人 | 审核修改 `req/index.md` |
| 5 | AI | `git diff req/` 查看改动 |
| 6 | AI | 理解修改意图，确认或讨论 |
| 7 | AI | `git commit` 更新 |

**核心原则**：先读再写，还原文档本意，不是机械转换

### 2. 需求审核 (req-review)

**目标**：从测试视角识别需求问题，生成问题清单

```
输入：req/index.md
输出：req/issues.md → 答案合并回 index.md
```

**双视角分析**：

| 视角 | 方法 | 关注点 |
|------|------|--------|
| 业务视角 | 518 分析法 | 5W（背景场景）+ 1H（功能需求）+ 8C（非功能约束） |
| 测试视角 | 六维扫描 | 前因后果、边界定义、异常场景、交互细节、数据状态、性能兼容 |

**问题分类**：

| 分类 | 标识 | 处理方式 |
|------|------|----------|
| 设计缺陷 | 🔴 Defect | 必须修复 |
| 描述问题 | 🟡 Issue | 需要澄清 |
| 建议 | 🟢 Suggestion | 可选采纳 |

**人机协作流程**：

| 步骤 | 执行者 | 动作 |
|------|--------|------|
| 1 | AI | 双视角分析需求 |
| 2 | AI | 生成 `req/issues.md` |
| 3 | AI | `git commit` |
| 4 | 人 | 填写问题答案 |
| 5 | AI | 合并答案到 `index.md` |
| 6 | AI | `git commit` |

### 3. 测试规划 (tc-plan)

**目标**：将需求转化为结构化的测试规划

```
输入：req/index.md
输出：tc/plan.md
```

**核心原则**：Scenario ≠ 数据

```
✅ 正确（3 个 Scenario）：
- 用户名密码登录
- 手机验证码登录
- 第三方登录

❌ 错误（这是 1 个 Scenario 的等价类）：
- 密码 <8 位
- 密码 8-20 位
- 密码 >20 位
```

**输出结构**：

```markdown
# Module: 用户认证

## Scenario: 用户名密码登录

### 测试关注点
- 密码错误次数限制（5次锁定）
- 密码长度边界（8-20位）
- 记住登录状态
```

### 4. 测试用例生成 (tc-gen)

**目标**：根据测试规划生成具体的测试用例

```
输入：tc/plan.md
输出：tc/{Module}/{Scenario}.md
```

**核心原则**：Case = 等价类的具体化

```
✅ 正确（从测试关注点展开）：
- [P1] 验证有效用户名和密码登录成功
- [P3][反向] 验证密码错误第5次后账户锁定
- [P4] 验证密码长度下边界（8位）登录成功

❌ 错误（简单复述）：
- 验证用户名密码登录功能
```

## 设计原则

### 1. 显式优于隐式

每个 SKILL.md 都明确定义：
- 输入输出
- 工作流程（Phase 1, 2, 3...）
- 检查清单
- 正反示例
- CRITICAL 约束

### 2. 闭环优于开环

每个阶段都有：
- 生成 → 审核 → 确认 → 提交 的完整闭环
- 明确的下一步引导（`/req-review`、`/tc-plan`、`/tc-gen`）

### 3. 协作优于自动化

- AI 生成初稿，人工审核修正
- Git 作为协作媒介，`git diff` 追踪变更
- 多轮对话，渐进式确认

### 4. 规范优于自由

- 统一的问题格式（4 字段）
- 统一的分类体系（六维扫描）
- 统一的优先级定义（critical/warning/info）

## 技术实现

### SKILL.md 结构

```yaml
---
name: skill-name
description: 技能描述和触发词
---

# 技能名称

## 核心原则
## 输入输出
## 工作流程
### Phase 1: ...
### Phase 2: ...
## 检查清单
## 正反示例
## CRITICAL
```

### CLI 工具

```bash
# 初始化需求目录
twu init <需求目录>

# 解析文档
twu parse <需求目录>

# 导出测试用例
twu export <需求目录> --format jsonl
```

### Git 协作模式

```bash
# AI 生成后
git commit -m "req-parse: 解析需求文档"

# 人工修改后，AI 查看改动
git diff req/

# AI 确认后
git commit -m "req-parse: 更新需求文档"
```

## 总结

TWU 的核心创新在于：

1. **上下文工程**：通过 SKILL.md 提供精确的任务上下文，而非依赖模糊的提示词
2. **HITL 设计**：每个阶段都有人工审核节点，AI 辅助而非替代
3. **Git 协作**：利用版本控制实现人机协作的变更追踪
4. **渐进式流程**：从原始文档到测试用例，分阶段确认，降低错误传播

这不是一个"AI 自动生成测试用例"的工具，而是一个**人机协作的测试工作流平台**。
