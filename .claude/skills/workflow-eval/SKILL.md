# 工作流评估（workflow-eval）

评估 TWU 工作流的执行质量：从会话记录、Git 提交、输出文件提取事实数据，然后**你来做真正的分析判断**，写出有依据的评分和改进建议。

## 核心原则

数据收集交给脚本，**分析判断是你的工作**。报告里每一个评分都需要文字依据，每一条改进建议都需要指向具体问题。不能留空，不能写占位符。

---

## 第一步：收集原始数据

运行数据收集脚本，获取执行事实：

```bash
# 自动检测当前项目最新会话（最常用）
uv run python eval_workflow.py --latest

# 指定 JSONL 会话文件
uv run python eval_workflow.py ~/.claude/projects/xxx/session.jsonl --type jsonl

# 指定 Git 提交范围
uv run python eval_workflow.py HEAD~5..HEAD --type git

# 指定输出目录（只评估产出文件）
uv run python eval_workflow.py 相框端-轮播展示/ --type dir
```

脚本会输出两个文件：
- `eval/{timestamp}/data.json` — 结构化原始数据
- `eval/{timestamp}/raw_data.md` — 可读摘要（供你快速浏览）

**如果脚本报错**，不要卡在这里。直接读 `tc/` 和 `req/` 目录，手动完成评估。

---

## 第二步：阅读输出内容

光看统计数字不够，你需要读实际内容：

- `req/index.md` — 需求是否完整、结构是否清晰？
- `tc/plan.md` — 模块划分是否合理、场景是否覆盖关键路径？
- 抽查 3-5 个测试用例文件 — 步骤是否可执行、预期结果是否明确？

同时运行格式验证：

```bash
uv run twu validate plan
uv run twu validate case
```

---

## 第三步：评估与打分

按三个维度评估，总分 100 分：

### 过程质量（30 分）

| 评分点 | 满分 | 评估方式 |
|--------|------|---------|
| Skill 调用顺序正确（req-parse→review→plan→gen） | 15 | 看 data.json 的 skills 字段 |
| 工具调用合理（无明显冗余） | 10 | 看 tools 统计，Write 次数是否异常多 |
| 用户交互适中（不需要频繁澄清） | 5 | 看 interactions 计数 |

### 输出质量（50 分）

| 评分点 | 满分 | 评估方式 |
|--------|------|---------|
| 需求覆盖完整（req/index.md 没有明显遗漏） | 20 | 实际阅读需求文档 |
| 测试用例内容准确可执行 | 20 | 抽查用例，步骤/预期结果是否够具体 |
| 格式通过验证工具 | 10 | validate 命令输出 |

### Git 提交规范（20 分）

| 评分点 | 满分 | 评估方式 |
|--------|------|---------|
| 提交信息规范（前缀正确、描述清晰） | 10 | 看 commits 列表 |
| 提交粒度合理（每个 skill 一次提交） | 10 | 看提交数量和内容对应关系 |

```bash

```

---

## 常见情况处理

**数据为空（tools/skills 全是 0）**：脚本可能没找到有效的会话文件。改用 `--type dir` 直接评估输出目录，或手动读文件。

**validate 报错**：记录具体错误信息，计入格式规范得分，同时在改进建议里指出。

**只有 Git 范围没有 JSONL**：用 `--type git` 模式，过程质量只评估 Git 提交部分，注明数据来源局限。
