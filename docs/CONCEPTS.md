# TWU 核心思想

> Test Workflow Utility —— 基于上下文工程的人机协作测试工作流

---

## 1. 上下文工程（Context Engineering）

传统的 Prompt Engineering 关注单次对话的提示词优化，而**上下文工程**关注的是为 AI 在执行任务时提供完整、结构化的上下文，使输出可预期、可复现。

```
传统方式：用户 → 模糊指令 → AI → 不确定的输出
上下文工程：用户 → 触发 Skill → SKILL.md 注入上下文 → AI → 规范化输出
```

三个关键要素：

| 要素 | 说明 |
|------|------|
| 结构化的任务定义 | 通过 SKILL.md 定义每个阶段的输入、输出、流程、检查清单 |
| 精确的上下文注入 | AI 执行任务时获得完整的领域知识和操作规范 |
| 可复现的工作流 | 相同的输入 + 相同的 SKILL = 一致的输出质量 |

### SKILL.md 结构

每个 Skill 文件遵循统一结构，明确定义以下内容：

```
## 核心原则     → 这个阶段的目标和判断标准
## 输入输出     → 前置依赖和产物定义
## 工作流程     → Phase 1、2、3... 分步执行
## 检查清单     → 完成标准，可逐项勾选
## 正反示例     → 什么是对的，什么是错的
## CRITICAL    → 不可违反的硬约束
```

---

## 2. HITL（Human-in-the-Loop）

**AI 不是替代人，而是辅助人。** 每个阶段都设计了人机协作的闭环，AI 生成初稿，人工审核修正，AI 理解改动并确认。

```
AI 生成 → 人工审核 → AI 理解改动 → 确认闭环 → 下一阶段
```

三个关键机制：

**Git 作为协作媒介**

AI 每次生成产物后立即 `git commit`，人工修改后 AI 通过 `git diff` 精确理解改动内容，而不是依赖用户的文字描述。

```bash
# AI 生成后
git commit -m "req-parse: 解析需求文档"

# 人工修改后，AI 读取改动
git diff req/
```

**显式的审核节点**

每个 Phase 结束都有明确的人工审核环节，AI 主动引导用户审核，不会自动跳过。

**渐进式确认**

不是一次性生成全部内容，而是分阶段生成、分阶段确认，错误不会在阶段间传播。

---

## 3. 设计原则

### 显式优于隐式

所有规则写在 SKILL.md 里，而不是依赖 AI 的"理解"。包括：输入输出、执行步骤、检查清单、正反示例。

### 闭环优于开环

每个阶段都有完整的 **生成 → 审核 → 确认 → 提交** 闭环，并有明确的下一步引导（`/req-review`、`/tc-plan`、`/tc-gen`）。

### 协作优于自动化

| 自动化工具 | TWU |
|------------|-----|
| AI 自动生成全部内容 | AI 生成初稿，人工逐阶段确认 |
| 一次执行，全部完成 | 分阶段执行，渐进式确认 |
| 输出结果难以追溯 | Git 记录每次变更，可回溯 |

### 规范优于自由

统一的格式（问题清单 4 字段、用例 5 字段）、统一的分类体系（六维扫描、优先级定义），降低 AI 输出的不确定性。

---

## 4. 为什么这样设计

TWU 的核心问题不是"AI 能不能生成测试用例"，而是：

- **隐性业务规则不可见**：AI 不了解业务，生成的用例歧义率高（实测约 27%）
- **需求本身不完备**：需求文档有缺失、矛盾、歧义，先修需求才能写好用例
- **人工审核成本高**：缺乏结构化产物，审核难以形成闭环

TWU 的回答是：**不跳过需求分析直接写用例**，而是通过 4 个阶段逐步建立共识，每个阶段都有人工确认节点，最终生成高质量的测试用例。
