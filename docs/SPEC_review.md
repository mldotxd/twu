# 需求问题规范

> 定义需求审核中"需求问题"的完整生命周期：发现 → 记录 → 评审 → 回答 → 闭环。

---

## 1. 输入要求

开始需求审核前，必须具备：

| 输入 | 来源 | 说明 |
|------|------|------|
| `req/index.md` | /req-parse 产出 | 标准化需求文档，经人工审核确认 |
| 业务背景理解 | 文档阅读 | 理解需求的业务目标、用户角色、使用场景 |

**不满足时**：提示先运行 `/req-parse` 完成需求解析。

---

## 2. 问题发现方法

### 2.1 业务视角：518 分析法

从业务全貌理解需求，使用 **5W + 1H + 8C** 框架：

**5W（背景与场景）**：

| 维度 | 关注点 |
|------|--------|
| Why | 为什么有这个需求？解决什么问题？ |
| Who | 用户是谁？利益相关者是谁？ |
| When | 有无时间范围限制？ |
| Where | 应用场景有哪些？ |
| What | 最终产出是什么？ |

**1H（功能性需求）**：

| 维度 | 关注点 |
|------|--------|
| How | 业务流程（正常/异常）、状态流转、已有流程兼容 |

**8C（非功能性约束）**：性能、效果、稳定性、可用性、兼容性、安全性、合规性、易用性。

### 2.2 测试视角：六维扫描

逐段阅读需求文档，从以下维度识别问题：

| 维度 | 关注点 | 示例 |
|------|--------|------|
| 前因后果 | 业务规则、权限约束、前置条件 | 选中需要什么权限？ |
| 边界定义 | 生效条件、范围排除、限制条件 | 最多选几个？ |
| 异常场景 | 失败处理、重试、回退、冲突 | 网络失败后如何处理？ |
| 交互细节 | 文案、提示、加载、空态、反馈 | 删除是否需要二次确认？ |
| 数据状态 | 字段定义、状态流转、数据约束 | 数据保留多久？ |
| 性能兼容 | 响应时间、并发、兼容范围 | 支持哪些浏览器？ |

**扫描思维链**：

```
阅读段落 → 假设要测试这段 → 能写出测试用例吗？ → 不能 → 记录问题
```

### 2.3 四大评估维度

| 维度 | 检查要点 |
|------|----------|
| 完整性 | 界面交互、权限权益、数据分析、文件处理是否定义清晰 |
| 合理性 | 设计是否符合常识和用户习惯，是否过度设计或不足 |
| 一致性 | 需求描述前后是否矛盾，与现有系统逻辑是否冲突 |
| 无歧义性 | 描述是否清晰，避免"可能、大概"等模糊词汇 |

### 2.4 问题筛选标准

**重点识别影响测试设计的需求问题**。

判断方法：问自己以下 4 个问题，如果答案是"不知道"且无法通过常识推断，应当提出。

| 判断问题 | 说明 |
|----------|------|
| 预期结果是什么？ | 不知道成功/失败的判断标准 |
| 边界值是什么？ | 不知道最大/最小/空值如何处理 |
| 操作流程是什么？ | 不知道具体步骤或分支逻辑 |
| 异常如何处理？ | 不知道失败/超时/冲突时的行为 |

**不应提出的问题**：

- 常识可推断的结果（按钮点击有反馈）
- 装饰性细节（颜色、动画样式）
- 纯业务背景（为什么做这个功能）
- 纯技术实现（用什么数据库）
- 文档已明确说明的内容

## 3. 问题记录规范

### 3.1 缺陷类型

对齐缺陷管理系统的分类：

| 缺陷类型 | 说明 | 示例 |
|----------|------|------|
| 需求描述缺失 | 该定义的内容没有写 | 未说明删除后数据如何处理 |
| 需求描述有歧义 | 写了但可有多种理解 | "支持批量操作"未说明具体操作范围 |
| 需求描述前后矛盾 | 文档内不同位置说法冲突 | 前文说自动重试，后文说等用户确认 |
| 需求或设计多余 | 过度设计、不必要的功能 | 简单列表页增加了不必要的拖拽排序 |
| 需求无法验证 | 无法设计测试用例来验证 | "系统应尽量快速响应"无具体指标 |
| 其他 | 不属于以上分类 | — |

### 3.2 优先级

| 优先级 | 说明 |
|--------|------|
| 紧急 | 不明确则无法设计测试用例 |
| 高 | 影响核心场景的用例设计 |
| 中 | 影响边界和异常用例 |
| 低 | 体验优化、可选改进 |

### 3.3 单条问题格式

每条问题包含以下字段，可直接映射到缺陷管理系统：

```markdown
### 问题 N: 简短标题

[缺陷类型] 需求描述缺失/需求描述有歧义/需求描述前后矛盾/需求或设计多余/需求无法验证/其他
[优先级] 紧急/高/中/低
[位置] 章节名称（或页码）

[问题描述]
需求文档提到"原文引用"，但没说清楚XXX，导致无法设计XXX相关的测试用例。

[需要澄清]
1. 具体问题1？
2. 具体问题2？

[产品回答]

---
```

**字段说明**：

| 字段 | 必填 | 说明 | 对应缺陷系统 |
|------|------|------|-------------|
| 标题 | 是 | 一句话概括问题 | 缺陷标题 |
| 缺陷类型 | 是 | 6 种分类之一 | 起因/缺陷类型 |
| 优先级 | 是 | 紧急/高/中/低 | 优先级 |
| 位置 | 是 | 问题在文档中的位置 | — |
| 问题描述 | 是 | 问题是什么、影响是什么 | 缺陷描述 |
| 需要澄清 | 是 | 具体要确认的问题列表 | — |
| 产品回答 | — | 留空给产品填写 | — |

## 4. 问题清单格式（issues.md）

```markdown
# 需求审核问题清单

> 来源文档：req/index.md
> 生成时间：YYYY-MM-DD
> 问题数量：共 N 个（紧急 X | 高 Y | 中 Z | 低 W）

---

## [章节名称1]

### 问题 1: 广告账户选中权限未定义

[缺陷类型] 需求描述缺失
[优先级] 紧急
[位置] 广告账户选中

[问题描述]
需求文档提到"点击某一个广告账户"，但没说清楚选中需要什么权限，无权限时如何处理。

[需要澄清]
1. 选中广告账户需要什么权限？
2. 没有权限时如何提示用户？

[产品回答]

---

## [章节名称2]

### 问题 2: 同步失败的重试逻辑前后矛盾

[缺陷类型] 需求描述前后矛盾
[优先级] 紧急
[位置] 数据同步

[问题描述]
前文说"失败后立即重试"，后文说"失败后等待用户确认"，逻辑矛盾。

[需要澄清]
1. 同步失败后是自动重试还是等待用户确认？
2. 如果自动重试，最多重试几次？

[产品回答]

---
```

**组织方式**：按需求文档的章节分组，章节内按优先级排序（紧急在前）。

## 5. 评审流程

### 5.1 生成问题清单

```
AI 分析需求文档（518 + 六维扫描）
  → 生成问题初稿
  → 自检（逐条审核，剔除不合格问题）
  → 写入 req/issues.md
  → git commit
  → 引导用户审核问题清单
```

### 5.2 自检（生成后、提交前）

生成问题清单后，逐条检查：

| 检查项 | 不通过则 |
|--------|----------|
| 这个问题影响测试用例设计吗？ | 删除 |
| 常识能推断出答案吗？ | 删除 |
| 文档中已经写清楚了吗？ | 删除 |
| 是纯业务背景/纯技术实现/装饰性细节吗？ | 删除 |
| 缺陷类型准确吗？ | 修正 |
| 优先级合理吗？ | 修正 |
| 有重复或重叠的问题吗？ | 合并 |
| 标题能一句话概括问题吗？ | 改写 |

自检完成后，才写入 issues.md 并提交。

### 5.3 用户审核问题清单

用户审核 issues.md 的内容：
- 问题是否合理（删除不必要的问题）
- 缺陷类型和优先级是否准确
- 是否有遗漏的问题需要补充

### 5.4 产品回答

**方式一：文件填写**

产品直接在 issues.md 的 `[产品回答]` 处填写。

**方式二：对话回复**

产品在对话中回答，AI 记录到 issues.md。

**回答检查**：
- 紧急问题必须回答
- 高优先级问题建议回答
- 中/低优先级问题可选回答

### 5.5 答案合并

将确认的答案合并到 `req/index.md`：

1. 在相关段落后补充说明
2. 使用 `<!-- 补充：来自 issues.md 问题N -->` 标注来源
3. 移除或更新 `[待确认]` 标记
4. issues.md 中已处理的问题标题后添加 `[已合并]`

```bash
git commit -m "req-review: 合并需求审核答案"
```

### 5.6 多轮迭代

合并后如果发现新问题，可继续追加到 issues.md，再次走回答→合并流程，直到需求文档完备。

```
发现问题 → 记录到 issues.md → 产品回答 → 合并到 index.md → 再次审核
    ^                                                         |
    └──────────────── 继续迭代（如有新问题）──────────────────┘
```

## 6. 产物

```
req/
├── index.md      # 需求文档（合并答案后更新）
└── issues.md     # 问题清单（记录所有问题和回答）
```

## 7. 正反示例

### 正确：需求描述缺失

```markdown
### 问题 1: 广告账户选中权限未定义

[缺陷类型] 需求描述缺失
[优先级] 紧急
[位置] 广告账户选中

[问题描述]
需求文档提到"点击某一个广告账户"，但没说清楚选中需要什么权限，无权限时如何处理，导致无法设计权限相关的测试用例。

[需要澄清]
1. 选中广告账户需要什么权限？
2. 没有权限时如何提示用户？

[产品回答]

---
```

**为什么正确**：缺少权限说明，直接影响测试用例设计

### 正确：需求描述前后矛盾

```markdown
### 问题 2: 同步失败的重试逻辑前后矛盾

[缺陷类型] 需求描述前后矛盾
[优先级] 紧急
[位置] 数据同步

[问题描述]
前文说"失败后立即重试"，后文说"失败后等待用户确认"，逻辑矛盾，无法确定预期行为。

[需要澄清]
1. 同步失败后是自动重试还是等待用户确认？
2. 如果自动重试，最多重试几次？

[产品回答]

---
```

**为什么正确**：前后矛盾，无法确定预期行为

### 正确：需求无法验证

```markdown
### 问题 3: 性能要求无具体指标

[缺陷类型] 需求无法验证
[优先级] 高
[位置] 性能要求

[问题描述]
需求文档提到"系统应尽量快速响应"，没有具体指标，无法设计性能测试用例。

[需要澄清]
1. 响应时间的具体要求是多少（如 P95 < 2s）？
2. 并发用户数的预期是多少？

[产品回答]

---
```

**为什么正确**："尽量快速"无法作为测试判定标准

### 错误：纯业务背景

```
问题：为什么要开发这个功能？竞品是怎么做的？
```

**为什么错误**：纯业务背景，不影响测试用例设计

### 错误：常识可判断

```
问题：点击保存按钮后是否有 loading 状态？
```

**为什么错误**：常识问题，按钮点击有反馈是基本交互规范

### 错误：过度细节

```
问题：加载动画是转圈还是进度条？颜色是什么？
```

**为什么错误**：装饰性细节，不影响功能测试
