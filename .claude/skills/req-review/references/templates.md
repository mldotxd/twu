# 常见问题类型模板

针对高频问题类型的标准格式模板，套用后替换占位符即可。

## 模板 A：边界值未定义

适用：需求中有数量/长度/时间等限制但未明确边界值。

```markdown
### 问题 N: [功能名称]的[参数名称]边界值未定义

[缺陷类型] 需求描述缺失
[优先级] 紧急（无法设计边界值用例）
[位置] [章节名称]

[问题描述]
需求文档提到"[原文引用]"，但没有明确[参数名称]的最小值、最大值和默认值，导致无法设计边界值测试用例。

[需要澄清]
1. [参数名称]的最小值是多少？
2. [参数名称]的最大值是多少？
3. 默认值是多少？
4. 超出范围时如何处理？（截断？拒绝？提示？）

[影响范围]
- [功能模块]：正常值、边界值（最小/最大）、越界值的测试用例均无法设计
- 输入校验模块：不知道校验规则

[参考]
边界值处理常见方式：
- 前端实时限制：输入超长时截断或禁止继续输入
- 后端校验：超出范围返回 400 错误码 + 错误描述
- 默认值设计：通常选取最常用的中间值
```

## 模板 B：异常处理未定义

适用：需求描述了正常流程，但未定义失败/超时/冲突等异常场景的处理。

```markdown
### 问题 N: [操作名称]失败时的处理逻辑未定义

[缺陷类型] 需求描述缺失
[优先级] 高
[位置] [章节名称]

[问题描述]
需求文档描述了"[操作名称]"的正常流程（"[原文引用]"），但没有说明失败时的处理逻辑，导致无法设计异常场景的测试用例。

[需要澄清]
1. [操作名称]失败时如何提示用户？（错误文案是什么？）
2. 失败后是否支持重试？重试次数上限？
3. 失败后数据是否回滚？（中间状态如何处理？）
4. 有无超时机制？超时时间是多少？

[影响范围]
- 异常处理模块：失败状态的 UI 表现和交互逻辑
- 数据一致性：失败后数据状态的验证

[参考]
常见的失败处理模式：
- Toast 提示 + 数据回滚（适合简单操作）
- 错误弹框 + 手动重试（适合关键操作）
- 静默重试 N 次（适合后台操作，失败才通知用户）
```

## 模板 C：状态转换条件未定义

适用：需求中有多种状态，但未明确状态之间的转换条件和触发时机。

```markdown
### 问题 N: [对象名称]从[状态A]切换到[状态B]的条件未定义

[缺陷类型] 需求描述缺失
[优先级] 紧急
[位置] [章节名称]

[问题描述]
需求文档提到[对象名称]存在"[状态A]"和"[状态B]"两种状态（"[原文引用]"），但没有明确从[状态A]切换到[状态B]的触发条件，导致无法设计状态转换的测试用例。

[需要澄清]
1. 什么条件触发从[状态A]→[状态B]的切换？
2. 切换是立即生效还是延迟生效？
3. 切换失败时的行为是什么？（状态回退？保持原状态？）
4. [状态A]和[状态B]之间是否可以双向切换？

[影响范围]
- [功能模块]：状态转换的正向和反向测试用例均无法设计
- 边界场景：触发切换条件的临界状态测试

[参考]
状态机设计常见模式：
- 有限状态机：明确每个状态的合法转换路径
- 状态持久化：切换后是否需要写入数据库
- 并发保护：同一对象被多个操作同时触发切换时如何处理
```
