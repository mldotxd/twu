# 需求审核问题示例

## 正确示例

### 正确：需求描述缺失

```markdown
### 问题 1: 广告账户选中权限未定义

[缺陷类型] 需求描述缺失
[优先级] 紧急
[位置] 广告账户选中

[问题描述]
需求文档提到"点击某一个广告账户"，但没说清楚选中需要什么权限，无权限时如何处理，导致无法设计权限相关的测试用例。

[需要澄清]
1. 选中广告账户需要什么权限？（只读权限？编辑权限？还是无限制？）
2. 没有权限时如何提示用户？（弹框？Toast？按钮置灰？）
3. 权限来自哪个角色体系？（账户角色、成员角色还是其他？）

[影响范围]
- 权限模块：权限判断和鉴权逻辑
- 账户选中场景：有权限 / 无权限的正反向测试用例均无法设计

[参考]
常见的权限不足处理方式：
- 操作前检查：按钮/入口置灰，hover 提示"无权限"
- 操作时检查：弹框提示"您无权执行此操作"
- 是否区分"无权限"和"账户不存在"两种状态

[产品回答]

---
```

**为什么正确**：缺少权限说明，直接影响测试用例设计；[需要澄清] 提供了思考维度，[影响范围] 说明了影响的模块，[参考] 提供了业界常见做法。

### 正确：需求描述前后矛盾

```markdown
### 问题 2: 同步失败的重试逻辑前后矛盾

[缺陷类型] 需求描述前后矛盾
[优先级] 紧急
[位置] 数据同步

[问题描述]
需求文档第 3 节提到"同步失败后立即重试"，第 7 节又说"同步失败后等待用户确认再重试"，两处描述相互矛盾，无法确定预期行为，导致无法设计同步失败相关的测试用例。

[需要澄清]
1. 同步失败后是自动重试还是等待用户确认？
2. 如果自动重试：最多重试几次？重试间隔是多少？
3. 如果等待确认：确认弹框文案是什么？用户取消后数据如何处理？

[影响范围]
- 数据同步模块：失败处理主流程
- 异常恢复模块：重试次数上限、退避策略
- 用户交互：是否需要设计确认弹框的测试场景

[参考]
常见的失败重试模式：
- 静默自动重试（通常 3 次），适合后台同步
- 用户感知重试（提示 + 手动触发），适合前台同步
- 两者结合：自动重试 N 次后提示用户介入

[产品回答]

---
```

**为什么正确**：引用了原文具体位置（第3节/第7节），[需要澄清] 覆盖了两种情况下的子问题，[影响范围] 明确了三个受影响的模块。

### 正确：需求无法验证

```markdown
### 问题 3: 性能要求无具体指标

[缺陷类型] 需求无法验证
[优先级] 高
[位置] 性能要求

[问题描述]
需求文档提到"系统应尽量快速响应"，没有具体的量化指标，无法设计性能测试用例，也无法判断测试结果是否达标。

[需要澄清]
1. 响应时间的具体要求是多少？（如 P95 < 2s，P99 < 5s）
2. 并发用户数的预期峰值是多少？
3. 是否有不同操作的差异化要求？（如搜索 < 1s，上传 < 30s）

[影响范围]
- 性能测试：无法设定通过/失败的判断阈值
- 压测方案：无法确定测试的目标并发量

[参考]
常见的性能指标定义方式：
- 接口响应：P95 < 500ms / P99 < 2s
- 页面加载：首屏 < 3s，交互响应 < 200ms
- 吞吐量：支持 X QPS 不降级

[产品回答]

---
```

**为什么正确**："尽量快速"无法作为测试判定标准；[需要澄清] 细化了三个必要维度，[参考] 提供了业界常用指标格式。

## 错误示例

```
问题：为什么要开发这个功能？竞品是怎么做的？
→ 错误：纯业务背景，不影响测试用例设计

问题：点击保存按钮后是否有 loading 状态？
→ 错误：常识问题，按钮点击有反馈是基本交互规范

问题：加载动画是转圈还是进度条？颜色是什么？
→ 错误：装饰性细节，不影响功能测试
```
