# 需求审核技能（req-review）

从测试视角审核 `req/index.md`，输出 `req/issues.md`，收集答案后合并回 `index.md`。

## 工作流程

流程总览：
1. 前置检查
2. 双视角分析
3. 生成并自检
4. 收集答案
5. 合并闭环

### 前置检查

1. `req/index.md` 必须存在，否则提示先运行 `/req-parse`
2. 检查 `req/issues.md` 是否存在：
   - 不存在 → 进入「双视角分析」
   - 已存在 → 询问用户意图（继续审核/重新生成/补充问题）

### 双视角分析

#### 业务视角：518 分析法

使用 5W + 1H + 8C 框架理解业务全貌：

| 维度 | 关注点 |
|------|--------|
| Why | 为什么有这个需求？解决什么问题？ |
| Who | 用户是谁？利益相关者是谁？ |
| When | 有无时间范围限制？ |
| Where | 应用场景有哪些？ |
| What | 最终产出是什么？ |
| How | 业务流程（正常/异常）、状态流转、已有流程兼容 |

8C（非功能性约束）：性能、效果、稳定性、可用性、兼容性、安全性、合规性、易用性。

#### 测试视角：六维扫描

逐段阅读 `req/index.md`，从 6 个维度识别问题：

| 维度 | 关注点 | 示例 |
|------|--------|------|
| 前因后果 | 业务规则、权限约束、前置条件 | 选中需要什么权限？ |
| 边界定义 | 生效条件、范围排除、限制条件 | 最多选几个？ |
| 异常场景 | 失败处理、重试、回退、冲突 | 网络失败后如何处理？ |
| 交互细节 | 文案、提示、加载、空态、反馈 | 删除是否需要二次确认？ |
| 数据状态 | 字段定义、状态流转、数据约束 | 数据保留多久？ |
| 性能兼容 | 响应时间、并发、兼容范围 | 支持哪些浏览器？ |

**扫描思维链**：

```
阅读段落 → 假设要测试 → 能写出测试用例吗？ → 不能则记录问题
```

#### 四大评估维度

| 维度 | 检查要点 |
|------|----------|
| 完整性 | 界面交互、权限权益、数据分析、文件处理是否定义清晰 |
| 合理性 | 设计是否符合常识和用户习惯，是否过度设计或不足 |
| 一致性 | 需求描述前后是否矛盾，与现有系统逻辑是否冲突 |
| 无歧义性 | 描述是否清晰，避免"可能、大概"等模糊词汇 |

#### 问题筛选

重点识别影响测试设计的需求问题。判断方法：

| 判断问题 | 说明 |
|----------|------|
| 预期结果是什么？ | 不知道成功/失败的判断标准 |
| 边界值是什么？ | 不知道最大/最小/空值如何处理 |
| 操作流程是什么？ | 不知道具体步骤或分支逻辑 |
| 异常如何处理？ | 不知道失败/超时/冲突时的行为 |

如果答案是"不知道"且无法通过常识推断，应当提出。

**不应提出的问题**：

- 常识可推断的结果（按钮点击有反馈）
- 装饰性细节（颜色、动画样式）
- 纯业务背景（为什么做这个功能）
- 纯技术实现（用什么数据库）
- 文档已明确说明的内容

### 生成并自检

#### 记录问题

缺陷类型（对齐缺陷管理系统）：

| 缺陷类型 | 说明 |
|----------|------|
| 需求描述缺失 | 该定义的内容没有写 |
| 需求描述有歧义 | 写了但可有多种理解 |
| 需求描述前后矛盾 | 文档内不同位置说法冲突 |
| 需求或设计多余 | 过度设计、不必要的功能 |
| 需求无法验证 | 无法设计测试用例来验证 |
| 其他 | 不属于以上分类 |

优先级：

| 优先级 | 说明 |
|--------|------|
| 紧急 | 不明确则无法设计测试用例 |
| 高 | 影响核心场景的用例设计 |
| 中 | 影响边界和异常用例 |
| 低 | 体验优化、可选改进 |

单条问题格式：

```markdown
### 问题 N: 简短标题

[缺陷类型] 需求描述缺失/需求描述有歧义/需求描述前后矛盾/需求或设计多余/需求无法验证/其他
[优先级] 紧急/高/中/低
[位置] 章节名称（或页码）

[问题描述]
需求文档提到"原文引用"，但没说清楚XXX，导致无法设计XXX相关的测试用例。

[需要澄清]
1. 具体问题1？
2. 具体问题2？

[影响范围]
- 影响的功能模块或测试场景

[参考]（可选）
- 类似场景的常见做法或思考方向（不是答案）

[产品回答]

---
```

**格式说明：**

1. **[问题描述]** - 必填
   - 引用需求原文
   - 说明不清楚的地方
   - 说明对测试的影响

2. **[需要澄清]** - 必填
   - 列出具体的问题点
   - 使用疑问句
   - 提供思考方向（如：最小值？最大值？默认值？）

3. **[影响范围]** - 必填
   - 说明这个问题会影响哪些功能模块
   - 说明会影响哪些测试场景
   - 帮助产品理解问题的重要性

4. **[参考]** - 可选
   - 提供类似场景的常见做法
   - 提供思考方向和参考
   - **注意：这不是答案，只是参考**

**示例：**

```markdown
### 问题 1: 文件有效性判断标准未定义

[缺陷类型] 需求描述缺失
[优先级] 紧急
[位置] 二、资源模型

[问题描述]
需求文档提到"文件有效"作为资源进入轮播列表的条件之一（"普通图片：已下载 + 未隐藏 + 文件有效"），但没有明确"文件有效"的判断标准，导致无法设计资源筛选相关的测试用例。

[需要澄清]
1. 文件是否存在？（File.exists()）
2. 文件大小是否>0？
3. 是否需要格式校验？（如图片格式、视频编码）
4. 文件损坏如何判断？是否需要解码校验？

[影响范围]
- 资源管理模块：资源进入轮播列表的判断逻辑
- 下载管理模块：下载失败后的重试和错误处理
- 异常处理模块：文件异常的处理策略

[参考]
在类似的资源管理场景中，通常会检查：
- 文件存在性：File.exists() 返回 true
- 文件大小：size > 0（避免空文件）
- 文件格式：扩展名匹配（.jpg/.mp4）
- 部分场景会做格式校验，但会影响性能

[产品回答]

---
```

#### 自检

生成问题后，逐条检查，提交前必须通过：

| 检查项 | 不通过则 |
|--------|----------|
| 这个问题影响测试用例设计吗？ | 删除 |
| 常识能推断出答案吗？ | 删除 |
| 文档中已经写清楚了吗？ | 删除 |
| 是纯业务背景/纯技术实现/装饰性细节吗？ | 删除 |
| 缺陷类型准确吗？ | 修正 |
| 优先级合理吗？ | 修正 |
| 有重复或重叠的问题吗？ | 合并 |
| 标题能一句话概括问题吗？ | 改写 |
| **[需要澄清]是否提供了具体的思考方向？** | **补充** |
| **[影响范围]是否说明了影响的模块和场景？** | **补充** |
| **[参考]是否提供了思考方向（而非答案）？** | **补充或删除** |

#### 写入并提交

1. 按章节分组，章节内按优先级排序（紧急在前）
2. 写入 `req/issues.md`（格式见「issues.md 模板」）
3. Git Commit：`req-review: 生成需求审核问题清单`
4. 引导用户：

```
问题清单已生成：req/issues.md

共 N 个问题（紧急 X | 高 Y | 中 Z | 低 W）

请审核并填写答案：
1. 打开 req/issues.md
2. 在每个问题的 [产品回答] 处填写
3. 完成后告知我合并

或直接在对话中回复答案。
```

**issues.md 模板**：

```markdown
# 需求审核问题清单

> 来源文档：req/index.md
> 生成时间：YYYY-MM-DD
> 问题数量：共 N 个（紧急 X | 高 Y | 中 Z | 低 W）

---

## [章节名称]

### 问题 1: 简短标题

[缺陷类型] ...
[优先级] ...
[位置] ...

[问题描述]
需求文档提到"原文引用"，但没说清楚XXX，导致无法设计XXX相关的测试用例。

[需要澄清]
1. 具体问题1（最小值？最大值？默认值？）
2. 具体问题2？

[影响范围]
- 影响的功能模块：...
- 影响的测试场景：...

[参考]（可选）
- 类似场景的常见做法：...

[产品回答]

---
```

组织方式：按需求文档的章节分组，章节内按优先级排序（紧急在前）。

### 收集答案

**方式一：文件填写**
- 产品在 `issues.md` 的 `[产品回答]` 处填写
- 完成后告知 Agent

**方式二：对话回复**
- 产品在对话中逐个回复
- Agent 记录并更新 `issues.md`

**回答检查**：
- 所有紧急问题必须回答
- 未回答的紧急问题 → 提示用户补充

**答案确认**：

```
收到以下答案，请确认：
- 问题 1: [答案摘要]
- 问题 2: [答案摘要]
...

确认后我将合并到 index.md。
```

### 合并闭环

#### 合并答案

将确认的答案合并到 `req/index.md`：
- 在相关段落后补充说明
- 使用 `<!-- 补充：来自 issues.md 问题N -->` 标注来源
- 移除或更新 `[待确认]` 标记

#### 更新 issues.md

已处理的问题标题后添加 `[已合并]`

#### 提交

```bash
git add req/
git commit -m "req-review: 合并需求审核答案"
```

#### 多轮迭代

合并后如发现新问题，继续追加到 issues.md，再次走回答→合并流程。

#### 引导下一步

```
需求审核完成：
- 问题清单：req/issues.md（X 个问题已处理）
- 需求文档：req/index.md（已更新）

下一步：/tc-plan 生成测试规划
```

## 正反示例

### 正确：需求描述缺失

```markdown
### 问题 1: 广告账户选中权限未定义

[缺陷类型] 需求描述缺失
[优先级] 紧急
[位置] 广告账户选中

[问题描述]
需求文档提到"点击某一个广告账户"，但没说清楚选中需要什么权限，无权限时如何处理，导致无法设计权限相关的测试用例。

[需要澄清]
1. 选中广告账户需要什么权限？（只读权限？编辑权限？还是无限制？）
2. 没有权限时如何提示用户？（弹框？Toast？按钮置灰？）
3. 权限来自哪个角色体系？（账户角色、成员角色还是其他？）

[影响范围]
- 权限模块：权限判断和鉴权逻辑
- 账户选中场景：有权限 / 无权限的正反向测试用例均无法设计

[参考]
常见的权限不足处理方式：
- 操作前检查：按钮/入口置灰，hover 提示"无权限"
- 操作时检查：弹框提示"您无权执行此操作"
- 是否区分"无权限"和"账户不存在"两种状态

[产品回答]

---
```

**为什么正确**：缺少权限说明，直接影响测试用例设计；[需要澄清] 提供了思考维度，[影响范围] 说明了影响的模块，[参考] 提供了业界常见做法。

### 正确：需求描述前后矛盾

```markdown
### 问题 2: 同步失败的重试逻辑前后矛盾

[缺陷类型] 需求描述前后矛盾
[优先级] 紧急
[位置] 数据同步

[问题描述]
需求文档第 3 节提到"同步失败后立即重试"，第 7 节又说"同步失败后等待用户确认再重试"，两处描述相互矛盾，无法确定预期行为，导致无法设计同步失败相关的测试用例。

[需要澄清]
1. 同步失败后是自动重试还是等待用户确认？
2. 如果自动重试：最多重试几次？重试间隔是多少？
3. 如果等待确认：确认弹框文案是什么？用户取消后数据如何处理？

[影响范围]
- 数据同步模块：失败处理主流程
- 异常恢复模块：重试次数上限、退避策略
- 用户交互：是否需要设计确认弹框的测试场景

[参考]
常见的失败重试模式：
- 静默自动重试（通常 3 次），适合后台同步
- 用户感知重试（提示 + 手动触发），适合前台同步
- 两者结合：自动重试 N 次后提示用户介入

[产品回答]

---
```

**为什么正确**：引用了原文具体位置（第3节/第7节），[需要澄清] 覆盖了两种情况下的子问题，[影响范围] 明确了三个受影响的模块。

### 正确：需求无法验证

```markdown
### 问题 3: 性能要求无具体指标

[缺陷类型] 需求无法验证
[优先级] 高
[位置] 性能要求

[问题描述]
需求文档提到"系统应尽量快速响应"，没有具体的量化指标，无法设计性能测试用例，也无法判断测试结果是否达标。

[需要澄清]
1. 响应时间的具体要求是多少？（如 P95 < 2s，P99 < 5s）
2. 并发用户数的预期峰值是多少？
3. 是否有不同操作的差异化要求？（如搜索 < 1s，上传 < 30s）

[影响范围]
- 性能测试：无法设定通过/失败的判断阈值
- 压测方案：无法确定测试的目标并发量

[参考]
常见的性能指标定义方式：
- 接口响应：P95 < 500ms / P99 < 2s
- 页面加载：首屏 < 3s，交互响应 < 200ms
- 吞吐量：支持 X QPS 不降级

[产品回答]

---
```

**为什么正确**："尽量快速"无法作为测试判定标准；[需要澄清] 细化了三个必要维度，[参考] 提供了业界常用指标格式。

### 错误示例

```
问题：为什么要开发这个功能？竞品是怎么做的？
→ 错误：纯业务背景，不影响测试用例设计

问题：点击保存按钮后是否有 loading 状态？
→ 错误：常识问题，按钮点击有反馈是基本交互规范

问题：加载动画是转圈还是进度条？颜色是什么？
→ 错误：装饰性细节，不影响功能测试
```

## 常见问题类型模板

针对高频问题类型，按照标准格式提供参考框架，直接套用并替换占位符。

### 模板 A：边界值未定义

适用：需求中有数量/长度/时间等限制但未明确边界值。

```markdown
### 问题 N: [功能名称]的[参数名称]边界值未定义

[缺陷类型] 需求描述缺失
[优先级] 紧急（无法设计边界值用例）
[位置] [章节名称]

[问题描述]
需求文档提到"[原文引用]"，但没有明确[参数名称]的最小值、最大值和默认值，导致无法设计边界值测试用例。

[需要澄清]
1. [参数名称]的最小值是多少？
2. [参数名称]的最大值是多少？
3. 默认值是多少？
4. 超出范围时如何处理？（截断？拒绝？提示？）

[影响范围]
- [功能模块]：正常值、边界值（最小/最大）、越界值的测试用例均无法设计
- 输入校验模块：不知道校验规则

[参考]
边界值处理常见方式：
- 前端实时限制：输入超长时截断或禁止继续输入
- 后端校验：超出范围返回 400 错误码 + 错误描述
- 默认值设计：通常选取最常用的中间值
```

### 模板 B：异常处理未定义

适用：需求描述了正常流程，但未定义失败/超时/冲突等异常场景的处理。

```markdown
### 问题 N: [操作名称]失败时的处理逻辑未定义

[缺陷类型] 需求描述缺失
[优先级] 高
[位置] [章节名称]

[问题描述]
需求文档描述了"[操作名称]"的正常流程（"[原文引用]"），但没有说明失败时的处理逻辑，导致无法设计异常场景的测试用例。

[需要澄清]
1. [操作名称]失败时如何提示用户？（错误文案是什么？）
2. 失败后是否支持重试？重试次数上限？
3. 失败后数据是否回滚？（中间状态如何处理？）
4. 有无超时机制？超时时间是多少？

[影响范围]
- 异常处理模块：失败状态的 UI 表现和交互逻辑
- 数据一致性：失败后数据状态的验证

[参考]
常见的失败处理模式：
- Toast 提示 + 数据回滚（适合简单操作）
- 错误弹框 + 手动重试（适合关键操作）
- 静默重试 N 次（适合后台操作，失败才通知用户）
```

### 模板 C：状态转换条件未定义

适用：需求中有多种状态，但未明确状态之间的转换条件和触发时机。

```markdown
### 问题 N: [对象名称]从[状态A]切换到[状态B]的条件未定义

[缺陷类型] 需求描述缺失
[优先级] 紧急
[位置] [章节名称]

[问题描述]
需求文档提到[对象名称]存在"[状态A]"和"[状态B]"两种状态（"[原文引用]"），但没有明确从[状态A]切换到[状态B]的触发条件，导致无法设计状态转换的测试用例。

[需要澄清]
1. 什么条件触发从[状态A]→[状态B]的切换？
2. 切换是立即生效还是延迟生效？
3. 切换失败时的行为是什么？（状态回退？保持原状态？）
4. [状态A]和[状态B]之间是否可以双向切换？

[影响范围]
- [功能模块]：状态转换的正向和反向测试用例均无法设计
- 边界场景：触发切换条件的临界状态测试

[参考]
状态机设计常见模式：
- 有限状态机：明确每个状态的合法转换路径
- 状态持久化：切换后是否需要写入数据库
- 并发保护：同一对象被多个操作同时触发切换时如何处理
```

## CRITICAL

- **重点识别需求问题**：缺失、歧义、矛盾、多余、无法验证
- **不提常识问题**：能通过常识推断的不提
- **先自检再提交**：生成问题后必须逐条自检，通过后才写入 issues.md
- **字段完整**：每条问题必须有以下字段：
  - 缺陷类型、优先级、位置（3 个元数据）
  - [问题描述]：引用原文 + 说明不清楚的地方 + 对测试的影响
  - [需要澄清]：具体疑问句 + 思考方向（最小值？最大值？默认值？）

  - [参考]：可选，仅提供思考方向，**不提供答案**
  - [产品回答]：留空待填
- **[需要澄清] 必须有思考方向**：不能只问问题，要给出思维锚点（例：是 A 还是 B？最大几次？）
- **[影响范围] 必须说明模块和场景**：帮助产品理解问题的测试影响范围
- **[参考] 不是答案**：只提供行业常见做法、思考角度，产品负责决策
