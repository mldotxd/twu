# 工作流评估技能（workflow-eval）

深度评估 skill 工作流的执行过程、输出质量和 trace，生成评估报告并提供改进建议。

## 核心原则

**评估 = 过程 + 结果 + 改进**

- 过程评估：分析执行历史、工具调用、时间消耗
- 结果评估：检查输出质量、格式正确性、内容完整性
- 改进建议：识别问题、提出优化方案

## 输入输出

| 项目 | 内容 |
|------|------|
| 输入 | 执行历史文件（.txt/.jsonl）/ git 记录 / 输出目录 |
| 输出 | `eval/{timestamp}/report.md` |

## 评估维度

### 1. 过程评估（Process）

**执行流程分析：**
- Skill 调用顺序是否正确
- 工具使用是否合理（Read/Write/Bash/Edit）
- 是否有冗余操作或重复调用
- 错误处理是否得当

**时间效率分析：**
- 总执行时间
- 各阶段耗时分布
- 是否有明显的性能瓶颈

**交互质量分析：**
- 用户交互次数
- 引导语是否清晰
- 是否有不必要的等待

### 2. 结果评估（Output）

**文档质量：**
- 结构完整性（章节、元数据）
- 格式正确性（Markdown 语法、标题层级）
- 内容准确性（是否符合需求）
- 可读性（语言表达、逻辑清晰）

**Git 提交质量：**
- 提交信息是否规范
- 提交粒度是否合理
- 文件变更是否符合预期

**工具输出验证：**
- `uv run twu validate` 是否通过
- 是否有格式错误或警告

### 3. Trace 分析（Trace）

**工具调用链：**
- 工具调用序列
- 参数传递是否正确
- 返回值处理是否完整

**状态转换：**
- 文件状态变化（创建/修改/删除）
- 数据流转是否清晰

**异常处理：**
- 是否有错误发生
- 错误恢复是否成功

## 工作流程

### 前置检查

1. 确认输入来源：
   - 执行历史文件路径（.txt/.jsonl）
   - 或指定 git 提交范围
   - 或指定输出目录进行静态分析

2. 创建评估目录：
   ```bash
   mkdir -p eval/{timestamp}
   ```

### 数据收集

#### 从执行历史收集

如果提供了执行历史文件（如 `docs/eval/01.txt`）：

1. 读取历史文件
2. 提取关键信息：
   - Skill 调用记录
   - 工具调用记录（Read/Write/Bash/Edit）
   - 用户交互记录
   - 时间戳信息
   - 错误信息

#### 从 Git 记录收集

如果指定了 git 提交范围：

```bash
# 获取提交列表
git log --oneline {start}..{end}

# 获取文件变更
git diff --stat {start}..{end}

# 获取详细变更
git diff {start}..{end}
```

#### 从输出目录收集

如果指定了输出目录（如 `相框端-轮播展示/`）：

1. 扫描目录结构
2. 读取关键文件：
   - `req/index.md`
   - `req/issues.md`
   - `tc/plan.md`
   - `tc/{Module}/{Scenario}.md`
3. 运行验证工具：
   ```bash
   uv run twu validate plan
   uv run twu validate case
   ```

### 生成改进方案

```markdown
# 改进方案

## 方案 1：优化用户交互时间

**目标：** 减少用户填写问题清单的时间

**实施步骤：**

1. 在 req-review 生成问题清单时，为每个问题提供：
   - 具体的示例
   - 参考答案（如果可以从代码或文档中推断）
   - 相关的代码片段或文档引用

2. 增加智能推荐：
   - 分析代码实现，自动填充部分答案
   - 标记高置信度的推荐答案

3. 优化问题描述：
   - 使用更清晰的语言
   - 提供多选项而非开放式问题

**预期效果：**
- 用户填写时间减少 30%
- 问题清单评分提升至 95/100

## 方案 2：增强文档质量检查

**目标：** 提升输出文档的质量

**实施步骤：**

1. 在 tc-plan 阶段增加验证：
   - 检查测试关注点是否具体（是否包含数值、条件）
   - 检查是否有模糊词汇（如"正常"、"合理"）

2. 在 tc-gen 阶段增加验证：
   - 检查预期结果是否可验证
   - 检查测试步骤和预期结果是否一一对应

3. 增加自动修正：
   - 识别常见问题并自动修正
   - 提供修正建议

**预期效果：**
- plan.md 评分提升至 95/100
- 用例评分提升至 92/100

## 方案 3：增加并行处理

**目标：** 减少总执行时间

**实施步骤：**

1. 在 tc-gen 阶段：
   - 并行生成多个用例文件
   - 使用多线程或异步处理

2. 在 req-parse 阶段：
   - 并行读取多个原始文档
   - 并行解构多个文档

**预期效果：**
- 总耗时减少 15%

## 实施优先级

1. **立即实施：** 方案 1（优化用户交互时间）
2. **短期实施：** 方案 2（增强文档质量检查）
3. **长期实施：** 方案 3（增加并行处理）
```
