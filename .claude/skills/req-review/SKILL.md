# 需求审核技能（req-review）

从测试视角审核 `req/index.md`，输出 `req/issues.md`，收集答案后合并回 `index.md`。

## 工作流程

流程总览：
1. 前置检查
2. 双视角分析
3. 生成并自检
4. 收集答案
5. 合并闭环

### 前置检查

1. `req/index.md` 必须存在，否则提示先运行 `/req-parse`
2. 检查 `req/issues.md` 是否存在：
   - 不存在 → 进入「双视角分析」
   - 已存在 → 询问用户意图（继续审核/重新生成/补充问题）

### 双视角分析

#### 业务视角：518 分析法

使用 5W + 1H + 8C 框架理解业务全貌：

| 维度 | 关注点 |
|------|--------|
| Why | 为什么有这个需求？解决什么问题？ |
| Who | 用户是谁？利益相关者是谁？ |
| When | 有无时间范围限制？ |
| Where | 应用场景有哪些？ |
| What | 最终产出是什么？ |
| How | 业务流程（正常/异常）、状态流转、已有流程兼容 |

8C（非功能性约束）：性能、效果、稳定性、可用性、兼容性、安全性、合规性、易用性。

#### 测试视角：六维扫描

逐段阅读 `req/index.md`，从 6 个维度识别问题：

| 维度 | 关注点 | 示例 |
|------|--------|------|
| 前因后果 | 业务规则、权限约束、前置条件 | 选中需要什么权限？ |
| 边界定义 | 生效条件、范围排除、限制条件 | 最多选几个？ |
| 异常场景 | 失败处理、重试、回退、冲突 | 网络失败后如何处理？ |
| 交互细节 | 文案、提示、加载、空态、反馈 | 删除是否需要二次确认？ |
| 数据状态 | 字段定义、状态流转、数据约束 | 数据保留多久？ |
| 性能兼容 | 响应时间、并发、兼容范围 | 支持哪些浏览器？ |

**扫描思维链**：

```
阅读段落 → 假设要测试 → 能写出测试用例吗？ → 不能则记录问题
```

#### 四大评估维度

| 维度 | 检查要点 |
|------|----------|
| 完整性 | 界面交互、权限权益、数据分析、文件处理是否定义清晰 |
| 合理性 | 设计是否符合常识和用户习惯，是否过度设计或不足 |
| 一致性 | 需求描述前后是否矛盾，与现有系统逻辑是否冲突 |
| 无歧义性 | 描述是否清晰，避免"可能、大概"等模糊词汇 |

#### 问题筛选

重点识别影响测试设计的需求问题。判断方法：

| 判断问题 | 说明 |
|----------|------|
| 预期结果是什么？ | 不知道成功/失败的判断标准 |
| 边界值是什么？ | 不知道最大/最小/空值如何处理 |
| 操作流程是什么？ | 不知道具体步骤或分支逻辑 |
| 异常如何处理？ | 不知道失败/超时/冲突时的行为 |

如果答案是"不知道"且无法通过常识推断，应当提出。

**不应提出的问题**：

- 常识可推断的结果（按钮点击有反馈）
- 装饰性细节（颜色、动画样式）
- 纯业务背景（为什么做这个功能）
- 纯技术实现（用什么数据库）
- 文档已明确说明的内容

### 生成并自检

#### 记录问题

缺陷类型（对齐缺陷管理系统）：

| 缺陷类型 | 说明 |
|----------|------|
| 需求描述缺失 | 该定义的内容没有写 |
| 需求描述有歧义 | 写了但可有多种理解 |
| 需求描述前后矛盾 | 文档内不同位置说法冲突 |
| 需求或设计多余 | 过度设计、不必要的功能 |
| 需求无法验证 | 无法设计测试用例来验证 |
| 其他 | 不属于以上分类 |

优先级：

| 优先级 | 说明 |
|--------|------|
| 紧急 | 不明确则无法设计测试用例 |
| 高 | 影响核心场景的用例设计 |
| 中 | 影响边界和异常用例 |
| 低 | 体验优化、可选改进 |

单条问题格式：

```markdown
### 问题 N: 简短标题

[缺陷类型] 需求描述缺失/需求描述有歧义/需求描述前后矛盾/需求或设计多余/需求无法验证/其他
[优先级] 紧急/高/中/低
[位置] 章节名称（或页码）

[问题描述]
需求文档提到"原文引用"，但没说清楚XXX，导致无法设计XXX相关的测试用例。

[需要澄清]
1. 具体问题1？
2. 具体问题2？

[产品回答]

---
```

#### 自检

生成问题后，逐条检查，提交前必须通过：

| 检查项 | 不通过则 |
|--------|----------|
| 这个问题影响测试用例设计吗？ | 删除 |
| 常识能推断出答案吗？ | 删除 |
| 文档中已经写清楚了吗？ | 删除 |
| 是纯业务背景/纯技术实现/装饰性细节吗？ | 删除 |
| 缺陷类型准确吗？ | 修正 |
| 优先级合理吗？ | 修正 |
| 有重复或重叠的问题吗？ | 合并 |
| 标题能一句话概括问题吗？ | 改写 |

#### 写入并提交

1. 按章节分组，章节内按优先级排序（紧急在前）
2. 写入 `req/issues.md`（格式见「issues.md 模板」）
3. Git Commit：`req-review: 生成需求审核问题清单`
4. 引导用户：

```
问题清单已生成：req/issues.md

共 N 个问题（紧急 X | 高 Y | 中 Z | 低 W）

请审核并填写答案：
1. 打开 req/issues.md
2. 在每个问题的 [产品回答] 处填写
3. 完成后告知我合并

或直接在对话中回复答案。
```

**issues.md 模板**：

```markdown
# 需求审核问题清单

> 来源文档：req/index.md
> 生成时间：YYYY-MM-DD
> 问题数量：共 N 个（紧急 X | 高 Y | 中 Z | 低 W）

---

## [章节名称]

### 问题 1: 简短标题

[缺陷类型] ...
[优先级] ...
[位置] ...

[问题描述]
...

[需要澄清]
1. ...

[产品回答]

---
```

组织方式：按需求文档的章节分组，章节内按优先级排序（紧急在前）。

### 收集答案

**方式一：文件填写**
- 产品在 `issues.md` 的 `[产品回答]` 处填写
- 完成后告知 Agent

**方式二：对话回复**
- 产品在对话中逐个回复
- Agent 记录并更新 `issues.md`

**回答检查**：
- 所有紧急问题必须回答
- 未回答的紧急问题 → 提示用户补充

**答案确认**：

```
收到以下答案，请确认：
- 问题 1: [答案摘要]
- 问题 2: [答案摘要]
...

确认后我将合并到 index.md。
```

### 合并闭环

#### 合并答案

将确认的答案合并到 `req/index.md`：
- 在相关段落后补充说明
- 使用 `<!-- 补充：来自 issues.md 问题N -->` 标注来源
- 移除或更新 `[待确认]` 标记

#### 更新 issues.md

已处理的问题标题后添加 `[已合并]`

#### 提交

```bash
git add req/
git commit -m "req-review: 合并需求审核答案"
```

#### 多轮迭代

合并后如发现新问题，继续追加到 issues.md，再次走回答→合并流程。

#### 引导下一步

```
需求审核完成：
- 问题清单：req/issues.md（X 个问题已处理）
- 需求文档：req/index.md（已更新）

下一步：/tc-plan 生成测试规划
```

## 正反示例

### 正确：需求描述缺失

```markdown
### 问题 1: 广告账户选中权限未定义

[缺陷类型] 需求描述缺失
[优先级] 紧急
[位置] 广告账户选中

[问题描述]
需求文档提到"点击某一个广告账户"，但没说清楚选中需要什么权限，无权限时如何处理，导致无法设计权限相关的测试用例。

[需要澄清]
1. 选中广告账户需要什么权限？
2. 没有权限时如何提示用户？

[产品回答]

---
```

**为什么正确**：缺少权限说明，直接影响测试用例设计。

### 正确：需求描述前后矛盾

```markdown
### 问题 2: 同步失败的重试逻辑前后矛盾

[缺陷类型] 需求描述前后矛盾
[优先级] 紧急
[位置] 数据同步

[问题描述]
前文说"失败后立即重试"，后文说"失败后等待用户确认"，逻辑矛盾，无法确定预期行为。

[需要澄清]
1. 同步失败后是自动重试还是等待用户确认？
2. 如果自动重试，最多重试几次？

[产品回答]

---
```

**为什么正确**：前后矛盾，无法确定预期行为。

### 正确：需求无法验证

```markdown
### 问题 3: 性能要求无具体指标

[缺陷类型] 需求无法验证
[优先级] 高
[位置] 性能要求

[问题描述]
需求文档提到"系统应尽量快速响应"，没有具体指标，无法设计性能测试用例。

[需要澄清]
1. 响应时间的具体要求是多少（如 P95 < 2s）？
2. 并发用户数的预期是多少？

[产品回答]

---
```

**为什么正确**："尽量快速"无法作为测试判定标准。

### 错误示例

```
问题：为什么要开发这个功能？竞品是怎么做的？
→ 错误：纯业务背景，不影响测试用例设计

问题：点击保存按钮后是否有 loading 状态？
→ 错误：常识问题，按钮点击有反馈是基本交互规范

问题：加载动画是转圈还是进度条？颜色是什么？
→ 错误：装饰性细节，不影响功能测试
```

## 检查清单

### 分析检查
- [ ] 518 分析：理解业务全貌
- [ ] 六维扫描：逐段阅读，无遗漏章节
- [ ] 每段都执行扫描思维链
- [ ] 四大维度：完整性、合理性、一致性、无歧义性

### 问题检查
- [ ] 逐条执行「自检」
- [ ] 每条问题影响测试用例设计
- [ ] 没有常识可推断的问题
- [ ] 没有纯业务背景、纯技术实现、装饰性问题
- [ ] 没有文档已明确说明的问题
- [ ] 没有重复或重叠的问题
- [ ] 每条问题的缺陷类型正确（6 种分类）
- [ ] 每条问题的优先级合理
- [ ] 每条问题有完整字段
- [ ] 按章节组织，章节内按优先级排序（紧急 > 高 > 中 > 低）

### 闭环检查
- [ ] 所有紧急问题已回答
- [ ] 答案合并到 index.md 相关段落
- [ ] issues.md 已处理的问题标记 `[已合并]`
- [ ] Git commit 完成
- [ ] 引导进入 /tc-plan

## CRITICAL

- **重点识别需求问题**：缺失、歧义、矛盾、多余、无法验证
- **不提常识问题**：能通过常识推断的不提
- **先自检再提交**：生成问题后必须逐条自检，通过后才写入 issues.md
- **字段完整**：每条问题必须有 缺陷类型、优先级、位置、问题描述、需要澄清、产品回答
- **完成后必须 commit**
