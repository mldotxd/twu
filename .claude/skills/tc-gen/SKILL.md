# 测试用例生成技能（tc-gen）

根据 `tc/plan.md` 生成具体的测试用例文件。

## 核心原则

**用例 = 等价类的具体化**

- Case 是 Scenario 测试关注点的等价类展开
- Case 不是 Scenario 的简单复述

```
正确（从测试关注点展开）：
Scenario: 用户名密码登录
测试关注点: 密码错误次数限制（5次）、密码长度边界（8-20位）

Case:
- [P1] 验证有效用户名和密码登录成功
- [P3][反向] 验证密码错误第5次后账户锁定
- [P4] 验证密码长度下边界（8位）登录成功
- [P4][反向] 验证密码长度不足（7位）登录失败

错误（简单复述）：
- 验证用户名密码登录功能
```

## 输入输出

| 项目 | 内容 |
|------|------|
| 输入 | `tc/plan.md` |
| 输出 | `tc/{Module}/{Scenario}.md` |

## 工具

| 命令 | 用途 | 来源 |
|------|------|------|
| `uv run twu validate case` | 检验用例文件格式 | 项目工具 |
| `uv run twu case-batch <xml>` | 批量修改/删除用例 | 项目工具 |

**validate 输出说明**：检测结果 + 引导语，根据引导语进行修复或自检。

**case-batch 操作类型**：
- `action="replace"` — 替换用例内容
- `action="delete"` — 删除用例

## 工作流程

流程总览：
1. 前置检查
2. 规划解析
3. 用例设计
4. 生成用例文件
5. 检验
6. 提交
7. 审核闭环

### 前置检查

1. `tc/plan.md` 必须存在，否则提示先运行 `/tc-plan`
2. 检查用例文件是否已存在：
   - 不存在 → 进入「规划解析」
   - 已存在 → 询问用户意图（继续审核/重新生成/补充用例）

### 规划解析

1. 读取 `tc/plan.md`
2. 提取所有 Module 和 Scenario
3. 对每个 Scenario，提取：
   - 风险等级（Critical/High/Medium/Low）
   - 测试关注点（具体的测试点列表）

### 用例设计（5 步推理）

对每个 Scenario 执行：

#### 理解场景

```
问题：这个场景的核心操作是什么？
输入：Scenario 名称 + 测试关注点

示例：
Scenario: 用户名密码登录
测试关注点: 密码错误次数限制（5次）、密码长度边界（8-20位）
核心操作: 用户输入用户名和密码进行登录
```

#### 识别等价类

从测试关注点提取等价类划分：

```
测试关注点 → 等价类

密码错误次数限制（5次）:
- 有效等价类: 错误次数 1-4 次
- 无效等价类: 错误次数 ≥5 次

密码长度边界（8-20位）:
- 有效等价类: 8-20 位
- 无效等价类: <8 位, >20 位
- 边界值: 7, 8, 20, 21 位
```

#### 设计用例

按功能重要性和测试类型设计用例。

#### 确定优先级

| 优先级 | 定义 |
|--------|------|
| P1 | 核心功能的正向用例（冒烟） |
| P2 | 基本功能的常用场景 + 核心功能的反向用例（重要错误和边界值） |
| P3 | 基本功能的不常用场景 |
| P4 | 基本功能的反向用例（异常测试、边界值） |
| P5 | 不常用功能的用例 |

**判断逻辑**：
1. 先判断功能重要性：核心功能 > 基本功能 > 不常用功能
2. 再判断测试类型：正向 > 反向
3. 核心功能的反向用例优先级高于基本功能的正向用例

#### 编写预期结果

**预期结果质量标准（CRITICAL）：**

预期结果必须满足以下5个标准：

1. **包含具体的状态变化**
   - ✓ "轮播间隔重置为10秒，从0开始计时"
   - ✗ "系统正常处理"

2. **包含可观察的现象**
   - ✓ "photo1.jpg出现在轮播列表中"
   - ✗ "资源加载成功"（如何观察？）

3. **避免模糊词汇**
   - 禁用词：正常、合理、适当、成功（单独使用）
   - ✓ "订单创建成功，订单号为ORD001，状态为待支付"
   - ✗ "订单创建成功"

4. **与测试步骤一一对应**
   - 3个测试步骤 → 3个预期结果
   - 每个步骤都有明确的预期

5. **可被测试人员验证**
   - 通过UI观察、日志查看、数据库查询等方式验证
   - ✓ "系统日志显示触发资源同步事件"
   - ✗ "内部状态更新"（无法观察）

**示例对比：**

```markdown
❌ 不合格的预期结果：
[预期结果] 1. 系统正常处理

✓ 合格的预期结果：
[预期结果] 1. photo2.jpg不在轮播列表中，系统日志显示"资源未下载，跳过加载"
```

#### 标记不确定点

无法确定的细节用 `[待确认]` 标记：

```markdown
[预期结果] 1. 订单创建成功，订单号为ORD001。2. [待确认] 库存扣减数量
```

### 生成用例文件

1. 确保 `tc/{Module}/` 目录存在
2. 每个 Scenario 生成一个 `.md` 文件
3. 文件格式：

```markdown
# {Scenario 名称}

> 所属模块: {Module}
> 风险等级: {Critical/High/Medium/Low}
> 用例数量: {N}

## [P1] 验证xxx
[测试类型] 功能
[前置条件] xxx
[测试步骤] 1. xxx
[预期结果] 1. xxx

## [P3][反向] 验证xxx
[测试类型] 功能
[前置条件] xxx
[测试步骤] 1. xxx
[预期结果] 1. xxx
```

### 检验

生成用例文件后，执行以下检验步骤：

#### 1. 格式检验

```bash
uv run twu validate case
```

- 有错误 → 修正后重新检验
- 全部通过 → 进入「预期结果质量自检」

#### 2. 预期结果质量自检

**自检清单（逐个用例检查）：**

对每个用例的预期结果，检查是否满足5个质量标准：

```
用例：[P1] 验证已下载且未隐藏的普通图片进入轮播列表
预期结果：1. photo1.jpg 出现在轮播列表中并正常展示

自检：
✓ 包含具体的状态变化？是（"出现在轮播列表中"）
✓ 包含可观察的现象？是（可以在UI上看到）
✗ 避免模糊词汇？否（"正常展示"是模糊词汇）
✓ 与测试步骤一一对应？是（1个步骤对应1个结果）
✓ 可被测试人员验证？是（通过UI观察）

结论：需要优化，去掉"正常展示"或改为具体描述
优化后：1. photo1.jpg 出现在轮播列表中，显示图片内容
```

**常见问题及修正：**

| 问题 | 示例 | 修正 |
|------|------|------|
| 使用模糊词汇 | "系统正常处理" | "photo2.jpg不在轮播列表中" |
| 缺少状态变化 | "资源加载成功" | "photo1.jpg下载完成，文件大小1.2MB" |
| 无法验证 | "内部状态更新" | "数据库中资源状态字段更新为1" |
| 不够具体 | "订单创建成功" | "订单创建成功，订单号ORD001，状态为待支付" |

**如果发现问题：**
1. 使用 Edit 工具修正预期结果
2. 重新运行 `uv run twu validate case`
3. 重新执行质量自检

**全部通过后 → 进入「提交」**

### 提交

1. 运行 `uv run twu validate case`，确保通过
2. Git Commit：`tc-gen: 生成测试用例`
3. 引导用户审核：

```
测试用例已生成：

tc/
├── 登录功能/
│   ├── 用户名密码登录.md (5 cases)
│   └── 手机验证码登录.md (4 cases)
└── 订单管理/
    └── 创建订单.md (6 cases)

请审核以下内容：
1. 用例覆盖是否完整
2. 测试数据是否合理
3. [待确认] 的点请确认或修改

审核完成后，可以：
- 直接修改用例文件
- 告诉我需要补充的用例
- 运行 twu case-batch 批量修改
```

### 审核闭环

#### 提取待确认点

如果用例中有 `[待确认]` 标记，生成审核清单：

```markdown
# 待确认点审核

## 订单管理 / 创建订单

> 文件：tc/订单管理/创建订单.md

### [P1] 验证正常创建订单

**完整用例**：
## [P1] 验证正常创建订单
[测试类型] 功能
[前置条件] 用户已登录，商品A库存充足
[测试步骤] 1. 选择商品A，数量1，点击下单
[预期结果] 1. 订单创建成功，[待确认] 库存扣减数量

**待确认点**：库存扣减数量

**确认结果**：
（用户填写）
```

#### 用户确认

用户可以：
- 直接修改用例文件
- 在审核清单中填写确认结果
- 口头告知 Agent

#### 批量修改（可选）

如果待确认点较多，Agent 生成批量操作文件：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<operations>
<case action="replace" file="tc/订单管理/创建订单.md" title="验证正常创建订单">
## [P1] 验证正常创建订单
[测试类型] 功能
[前置条件] 用户已登录，商品A库存充足
[测试步骤] 1. 选择商品A，数量1，点击下单
[预期结果] 1. 订单创建成功，库存扣减1
</case>
</operations>
```

执行批量修改：

```bash
uv run twu case-batch operations.xml
```

#### 完成检查

1. 查看修改：`git diff tc/`
2. 检查状态：
   - 还有 `[待确认]` → 继续确认
   - 无 `[待确认]` → 用例完成
3. 提交更新：`tc-gen: 更新测试用例`

- [ ] 用例标题以"验证"开头

- [ ] 前置条件明确（或省略表示无特殊前置）

  - [ ] 与测试步骤一一对应（数量一致）

## 用例格式规范

### 用例结构

```markdown
## [优先级] 用例标题
[测试类型] 类型值
[前置条件] 前置条件描述（可省略）
[测试步骤] 1. 步骤1。2. 步骤2。3. 步骤3
[预期结果] 1. 结果1。2. 结果2。3. 结果3
```

反向用例添加 `[反向]` 标记：`## [P2][反向] 验证xxx`

### 字段规范

**用例标题**：以"验证"开头，描述测试目的

**测试类型**（12 种枚举，必须完全匹配）：
- 功能、兼容性、易用性、性能、稳定性、安全性
- 可靠性、效果（AI类、资源类）、效果（硬件器件类）
- 可维护性、可移植性、埋点

**前置条件**：测试执行前的准备工作，无特殊前置可省略

**测试步骤**：
- 编号从 1 开始，中文句号分隔
- 每个步骤到达一个验证点
- 数据必须具体（test_user，不是 {username}）

**预期结果**：
- 编号从 1 开始，中文句号分隔
- **数量必须与测试步骤一致**
- 必须可验证，避免含糊词（单独的"成功"、"正常"）

## 示例

### 示例 1：从 plan.md 生成用例

**输入 plan.md**：
```markdown
## 登录功能

### 用户名密码登录
> 风险等级: Critical
> 测试关注点: 密码错误次数限制（5次）、密码长度边界（8-20位）
```

**输出 用户名密码登录.md**：
```markdown
# 用户名密码登录

> 所属模块: 登录功能
> 风险等级: Critical
> 用例数量: 5

## [P1] 验证有效用户名和密码登录成功
[测试类型] 功能
[前置条件] 已注册用户test_user，密码Test1234
[测试步骤] 1. 输入用户名test_user，输入密码Test1234，点击登录
[预期结果] 1. 登录成功，跳转到首页

## [P3][反向] 验证密码错误第5次后账户锁定
[测试类型] 功能
[前置条件] 已注册用户test_user，密码Test1234
[测试步骤] 1. 输入用户名test_user，连续5次输入错误密码wrong123，点击登录
[预期结果] 1. 第5次登录失败后，提示"账户已锁定，请30分钟后重试"

## [P4] 验证密码长度下边界（8位）登录成功
[测试类型] 功能
[前置条件] 已注册用户test_user，密码Abcd1234（8位）
[测试步骤] 1. 输入用户名test_user，输入密码Abcd1234，点击登录
[预期结果] 1. 登录成功，跳转到首页

## [P4][反向] 验证密码长度不足（7位）登录失败
[测试类型] 功能
[前置条件] 进入登录页
[测试步骤] 1. 输入用户名test_user，输入密码Abc1234（7位），点击登录
[预期结果] 1. 登录失败，提示"密码长度需8-20位"

## [P4][反向] 验证密码长度超限（21位）登录失败
[测试类型] 功能
[前置条件] 进入登录页
[测试步骤] 1. 输入用户名test_user，输入密码Abcdefghij1234567890X（21位），点击登录
[预期结果] 1. 登录失败，提示"密码长度需8-20位"
```

## CRITICAL

- **用例 = 等价类展开**：从测试关注点提取等价类，不是简单复述场景
- **测试数据必须具体**：使用真实数据，不用占位符
- **步骤与结果一一对应**：测试步骤和预期结果数量必须一致
- **预期结果必须满足5个质量标准**：
  1. 包含具体的状态变化
  2. 包含可观察的现象
  3. 避免模糊词汇（正常、合理、适当、成功单独使用）
  4. 与测试步骤一一对应
  5. 可被测试人员验证
- **生成后必须执行预期结果质量自检**：逐个用例检查，发现问题立即修正
- **提交前必须检验**：运行 `uv run twu validate case`，确保通过
- **完成后必须 commit**：`tc-gen: 生成测试用例` 或 `tc-gen: 更新测试用例`
- **审核闭环**：通过 git diff 查看用户修改，理解确认结果
