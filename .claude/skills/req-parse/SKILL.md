# 需求解析技能（req-parse）

阅读理解原始文档，还原文档本意，输出 `req/index.md`。

## 核心原则

**先读再写**：重新理解并生成标准化文档，不是简单复制或格式转换。

## 输入输出

| 项目 | 内容 |
|------|------|
| 输入 | `raw/`（PDF/Word/图片/Markdown/混合内容） |
| 输出 | `req/index.md` |

## 工具

| 命令 | 用途 | 来源 |
|------|------|------|
| `uv run twu parse <目录>` | 解构文档为 chunks/ + assets/ | 项目工具 |

## 工作流程

流程总览：
1. 输入识别
2. 图片理解
3. 阅读理解与标准化
4. 提交
5. 审核闭环

### 输入识别

#### 识别内容类型

| 类型 | 处理方式 |
|------|----------|
| PDF (.pdf) | twu parse → Docling 解析 → chunks/ + assets/ |
| Word (.docx) | twu parse → Docling 解析 → chunks/ + assets/ |
| Word (.doc) | **不支持**，提示用户转换为 .docx |
| Markdown (.md) | twu parse → 直接复制到 chunks/（仍需理解和标准化） |
| 图片 (.png/.jpg/.gif/.webp) | twu parse → 复制到 assets/ + 创建占位符 → 「图片理解」多模态分析 |
| Excel/PPT/HTML 等 | **不支持**，提示用户导出为 PDF 或截图 |
| 混合 | 分别处理后合并 |

#### 脚本解构

```bash
uv run twu parse <需求目录>
```

输出：`req/chunks/`（文本分片）、`req/assets/`（图片）、图片占位符 `<!-- image: xxx.png -->`

#### 兜底方案

Docling 失败时：PDF → PyPDF2，Word → python-docx，标记 `[解析质量待确认]`

### 图片理解

#### 图片分类思维链

```
观察特征 → 判断类型 → 针对性分析

类型判断：
- UI 元素（按钮、输入框）→ 原型图/界面截图
- 流程箭头、连接线 → 流程图
- 行列结构 → 表格截图
- 组件层级 → 架构图
- 图表元素 → 数据图表
- 概念性内容 → 说明图
```

#### 分析要点

| 图片类型 | 必须提取 |
|---------|---------|
| 原型图 | 所有可见文字、按钮文案、placeholder、提示信息、布局结构 |
| 流程图 | 步骤、分支条件、连接关系、判断逻辑 |
| 表格 | 表头、行列结构、数据内容 → Markdown 表格 |
| 架构图 | 组件名称、层级关系、数据流向 |

#### 辅助信息

结合图片名称理解：`login_page.png` → 登录页面，`order_flow.png` → 订单流程

### 阅读理解与标准化

**核心**：还原文档本意，输出连贯完整的文档，不是机械拼接。

#### 合并理解思维链

```
Step 1: 全局扫描
  有多少 chunks/图片？→ 覆盖哪些主题？→ 有无重叠或缺失？

Step 2: 建立文档骨架
  从 chunks 中识别章节结构 → 确定标题层级 → 排列逻辑顺序
  （多个 chunk 可能属于同一章节，需要合并）

Step 3: 逐章填充
  每个章节 → 找到对应的 chunks + 图片分析结果
  → 用自己的理解重写（不是拼接原文）
  → 图片描述融入正文上下文（不是孤立插入）

Step 4: 交叉验证
  文字描述和图片内容是否一致？
  → 不一致则标记 [待确认：文字与图片描述不一致]
  前后章节逻辑是否连贯？
  → 缺过渡则补充上下文

Step 5: 补缺标记
  哪些信息只有图片没有文字说明？→ 基于图片理解补充
  哪些信息无法确定？→ 标记 [待确认]
```

#### 不同输入场景的处理

**大 PDF（多 chunks）**：

```
chunks 可能存在：
- 跨页断句 → 识别并拼接完整段落
- 重复页眉页脚 → 识别并去除
- 表格跨页 → 合并为完整表格
- 章节被拆到多个 chunk → 按语义合并

关键：先理解整体结构，再按章节组织，不按 chunk 顺序堆叠
```

**纯截图（无文字 chunks）**：

```
全部信息来自图片理解结果 →
按图片之间的逻辑关系组织文档结构 →
图片名称/编号辅助判断顺序 →
每张图的分析结果扩写为完整段落
```

**混合输入（PDF + 截图 + Markdown）**：

```
不同来源可能描述同一功能 →
以文字描述为骨架，图片描述为补充 →
信息冲突时标记 [待确认：来源A说xxx，来源B说yyy]
```

#### 标准化输出

1. 统一标题层级
2. 整理段落结构
3. 生成 `req/index.md`，添加来源标注 `<!-- 来源：xxx.md -->`

### 提交

1. 写入 `req/index.md`
2. Git Commit：`req-parse: 解析需求文档`
3. 引导用户审核：

```
需求文档已解析：req/index.md

请审核：
1. 文档结构是否完整
2. 图片描述是否准确
3. [待确认] 的点请确认或修改

直接修改文件，完成后告知我。
```

### 审核闭环

**用户审核**：直接修改 `req/index.md`，完成后告知

**查看用户改动**：
```bash
git diff req/
```

根据 diff 理解修改 → 确认或讨论 → 用户确认 → `req-parse: 更新需求文档` → 引导 `/req-review`

## 产物结构

```
req/
├── index.md      # 最终产物
├── chunks/       # 中间产物
└── assets/       # 图片资源
```

## CRITICAL

- **先读再写**：重新理解并生成，不是复制
- **图片分类思维链**：先分类再针对性分析
- **Asset 提取严格**：确保所有图片正确处理
- **还原文档本意**：标准化的核心是理解
- **完成后必须 commit**
- **引导用户**：明确下一步操作
