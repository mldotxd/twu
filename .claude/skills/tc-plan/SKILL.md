# 测试规划技能（tc-plan）

将需求文档转化为结构化的测试规划，输出 `tc/plan.md`。

## 核心原则

**场景 ≠ 数据**

- Scenario 是独立的操作路径
- Scenario 不是不同的测试数据

```
正确（3 个 Scenario）：
- 用户名密码登录
- 手机验证码登录
- 第三方登录

错误（这是 1 个 Scenario 的等价类）：
- 密码 <8 位
- 密码 8-20 位
- 密码 >20 位
```

## 输入输出

| 项目 | 内容 |
|------|------|
| 输入 | 需求文档目录 |
| 输出 | `tc/plan.md` |

## 工具

| 命令 | 用途 | 来源 |
|------|------|------|
| `uv run twu validate plan` | 检验 plan.md 格式 | 项目工具 |

**validate 输出说明**：检测结果 + 引导语，根据引导语进行修复或自检。

## 工作流程

流程总览：
1. 前置检查
2. 需求分析
3. 结构规划
4. 生成 plan.md
5. 检验
6. 提交
7. 审核闭环

### 前置检查

检查 `tc/plan.md` 是否存在：
- 不存在 → 进入「需求分析」
- 已存在 → 询问用户意图（继续审核/重新生成/补充场景）

### 需求分析

1. 读取需求文档目录中的所有文件
2. 理解业务范围，识别功能模块
3. 如有不清晰的地方，向用户确认

**需求模糊时**：先输出 suggestion 让用户确认，而不是直接生成 plan.md：

```
根据需求文档，我理解的测试范围如下：

Module:
1. 登录功能
2. 订单管理

Scenario:
- 登录功能：用户名密码登录、手机验证码登录
- 订单管理：创建订单、取消订单

请确认这个理解是否正确，或者需要调整？
```

### 结构规划（5 步推理）

#### 理解业务范围

识别 Module（业务模块）。

```
问题：这个需求涉及哪些业务模块？

示例：
- 登录功能
- 订单管理
- 用户设置
```

#### 识别操作路径

识别 Scenario（独立场景）。

```
判断标准：
- 是否是独立的用户操作流程？
- 是否有不同的前置条件或触发条件？
- 是否有不同的业务规则？

示例：
Module: 登录功能
Scenario:
- 用户名密码登录
- 手机验证码登录
- 第三方登录
```

#### 避免数据混淆

检查：这些是真的不同场景，还是同一场景的不同数据？

```
错误：密码长度 <8、8-20、>20（这是等价类，属于 1 个 Scenario）
正确：用户名密码登录、手机验证码登录（这是不同操作路径）
```

#### 评估风险

为每个 Scenario 标记风险等级。

| 等级 | 定义 | 示例 |
|------|------|------|
| Critical | 核心业务流程 | 用户登录、订单创建 |
| High | 重要功能 | 手机验证码登录、订单查询 |
| Medium | 辅助功能 | 修改收货地址、第三方登录 |
| Low | 边缘功能 | 特殊字符处理 |

#### 提炼测试关注点

从以下维度提炼（必须具体，有数值）：

- 边界值：长度、数量、时间限制
- 异常情况：失败、超时、冲突
- 业务规则：权限、状态、约束
- 数据约束：格式、范围、依赖

```
正确：验证码过期（60s）、错误次数限制（5次）
错误：验证码验证、登录失败处理
```

### 生成 plan.md

按以下格式生成 `tc/plan.md`：

```markdown
# 测试规划

> 需求来源: {需求文档目录}
> 生成时间: {当前日期}

## {Module 名称}

### {Scenario 名称}
> 风险等级: {Critical/High/Medium/Low}
> 测试关注点: {具体的测试点，用顿号分隔}
```

**[待确认] 标记**：需求不明确的点用 `[待确认]` 标记，可标记在 Scenario 标题或测试关注点中：

```markdown
### [待确认] 手机验证码登录
> 风险等级: High
> 测试关注点: [待确认] 验证码过期时间、[待确认] 错误次数限制
```

### 检验

```bash
uv run twu validate plan
```

- 有错误 → 修正后重新检验
- 全部通过 → 进入「提交」

### 提交

1. 确保 `tc/` 目录存在
2. 写入 `tc/plan.md`
3. Git Commit：`tc-plan: 生成测试规划`
4. 引导用户审核：

```
测试规划已生成：tc/plan.md

请审核以下内容：
1. Module/Scenario 划分是否合理
2. 风险等级是否准确
3. [待确认] 的点请确认或修改

审核完成后，可以：
- 直接修改 plan.md 文件
- 告诉我需要补充的场景
- 运行 /tc-gen 生成测试用例
```

### 审核闭环

#### 用户审核

用户可以：
- 直接修改 plan.md 文件
- 口头告知 Agent 需要修改的内容
- 确认或修改 `[待确认]` 标记的内容

#### 查看修改

```bash
git diff tc/plan.md
```

分析用户修改内容，理解意图：
- 确认：`[待确认] 验证码过期时间` → `验证码过期（60s）`
- 修改：调整风险等级、测试关注点
- 删除：移除不需要的 Scenario
- 补充：添加新的 Module 或 Scenario

#### 完成检查

1. 检查是否还有 `[待确认]` 标记
   - 有 → 提示用户继续确认
   - 无 → 规划完成
2. 提交更新：`tc-plan: 更新测试规划`
3. 引导进入 tc-gen：

```
检测到 plan.md 已无 [待确认] 标记。

测试规划已完成，可以运行 /tc-gen 生成测试用例。
```

## 检查清单

### 结构检查
- [ ] 每个 Module 至少有 1 个 Scenario
- [ ] Scenario 是独立操作路径，不是数据差异
- [ ] 没有将等价类拆分为多个 Scenario

### 内容检查
- [ ] 每个 Scenario 有风险等级
- [ ] 测试关注点具体（有数值、有条件）
- [ ] 不确定的点已标记 [待确认]

### 格式检查（脚本检验）
- [ ] Module 使用 `##` 标题
- [ ] Scenario 使用 `###` 标题
- [ ] 元数据使用 `>` 引用块
- [ ] 风险等级在 Critical/High/Medium/Low 内
- [ ] Scenario 名称不重复

### 闭环检查
- [ ] `uv run twu validate plan` 通过
- [ ] Git commit 完成
- [ ] 引导用户审核或进入 /tc-gen

## 示例

### 示例 1：首次生成

```
用户：/tc-plan requirements/
```

输出 plan.md 后引导用户审核。

### 示例 2：用户补充场景

```
用户：还需要加一个"忘记密码"的场景
```

直接在 plan.md 的登录功能 Module 下添加：

```markdown
### 忘记密码
> 风险等级: High
> 测试关注点: [待确认] 重置方式（邮箱/手机）、重置链接有效期、新密码规则
```

### 示例 3：审核完成

```
用户修改 plan.md，删除所有 [待确认] 标记后再次调用

Agent 检测到无 [待确认]，引导：
"测试规划已完成，可以运行 /tc-gen 生成测试用例。"
```

## CRITICAL

- **场景 ≠ 数据**：Scenario 是操作路径，不是等价类
- **测试关注点必须具体**：有数值、有条件，不能模糊
- **提交前必须检验**：运行 `uv run twu validate plan`，确保通过
- **完成后必须 commit**：`tc-plan: 生成测试规划` 或 `tc-plan: 更新测试规划`
- **审核闭环**：通过 git diff 查看用户修改，理解确认结果
- **引导用户**：明确告知下一步操作
