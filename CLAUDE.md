# TWU 项目背景

> 本文件是 Claude Code 的项目级知识背景，在每次会话时自动加载。

## 项目定位

TWU（Test Workflow Utility）是一套测试工作流技能包 + CLI 工具链，运行在 Claude Code 上。

核心解决的问题：**需求不完备 → 用例歧义率高**。不跳过需求分析直接写用例，而是分四个阶段逐步建立共识，每个阶段都有人工确认节点。

## 四阶段工作流

```
/req-parse → /req-review → /tc-plan → /tc-gen
（需求解析）  （需求审核）  （测试规划）（用例生成）
```

每个阶段：AI 生成 → git commit → 人工审核 → 确认 → 下一阶段

## 核心概念

### 三层测试结构

| 层级 | 定义 | 示例 |
|------|------|------|
| Module | 业务模块 | 登录功能、订单管理 |
| Scenario | 独立操作路径（不是数据差异） | 用户名密码登录、手机验证码登录 |
| Case | 等价类的具体化 | [P1] 验证有效用户名和密码登录成功 |

### 用例优先级

| 优先级 | 定义 |
|--------|------|
| P1 | 核心功能正向流程 |
| P2 | 基本功能正向流程 |
| P3 | 核心功能异常场景 |
| P4 | 边界条件 |
| P5 | 低频场景 |

### 需求问题优先级

| 优先级 | 说明 |
|--------|------|
| 紧急 | 不明确则无法设计测试用例 |
| 高 | 影响核心场景的用例设计 |
| 中 | 影响边界和异常用例 |
| 低 | 体验优化、可选改进 |

## 需求审核方法

**518 分析法**：5W（背景场景）+ 1H（功能需求）+ 8C（非功能约束）

**六维扫描**：前因后果、边界定义、异常场景、交互细节、数据状态、性能兼容

**问题筛选标准**：这个问题是否影响测试用例设计或测试结果判断？影响则记录，否则过滤。

## 目录约定

```
{需求目录}/
├── raw/          # 原始文档（输入）
├── req/
│   ├── index.md  # 标准化需求文档
│   └── issues.md # 需求问题清单
└── tc/
    ├── plan.md   # 测试规划
    └── {Module}/{Scenario}.md  # 测试用例
```

## 执行规范

- Python 使用 `uv run python`，禁止直接调用 `python` / `python3`
- 每个阶段完成后必须 `git commit`
- `[待确认]` 标记需用户确认后清空，不得残留
- Scenario 按操作路径划分，不是数据差异
