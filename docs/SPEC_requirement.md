# 需求流程规范

## 流程总览

```
┌─────────────────────────────────────────────────────────────┐
│ req-parse: 解构与标准化                                      │
├─────────────────────────────────────────────────────────────┤
│ 1. 脚本解构原始文档 → chunks/                                │
│ 2. 阅读理解 → 转成标准文档（还原本意）                        │
│ 3. git commit 标记                                          │
│ 4. 用户 diff 查看 → 审核修正（多轮对话）                     │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ req-review: 测试与需求完备性                                 │
├─────────────────────────────────────────────────────────────┤
│ 1. 产出需求问题文档 (issues.md)                              │
│ 2. 产品确认答案                                              │
│ 3. 补充到原文档                                              │
│ 4. 审核完备性（多轮交互）                                    │
│ 5. 最终形成完备的 REQ                                        │
└─────────────────────────────────────────────────────────────┘
```

**核心原则**：先读再写，不是简单复制，而是重新理解并生成标准化文档。

---

## Skill 1: req-parse（解构与标准化）

### 目标

阅读理解原始文档，还原文档本意，生成标准化的需求文档。

### 输入输出

| 项目 | 内容 |
|------|------|
| 输入 | `raw/`（PDF/Word/图片/Markdown/混合内容） |
| 输出 | `req/index.md` |

### 工作流程

#### Phase 0: 前置检查

1. 检查 `raw/` 目录是否存在
   - 不存在 → 提示用户创建并上传文档
2. 检查 `req/index.md` 是否存在
   - 不存在 → 进入 Phase 1
   - 已存在 → 询问用户意图（继续审核/重新解析）

#### Phase 1: 输入识别与解构

**1.1 扫描 raw/ 目录，识别内容类型**

| 内容类型 | 识别方式 | 处理方式 |
|---------|---------|---------|
| PDF | `.pdf` 扩展名 | Docling 解析 |
| Word | `.docx`/`.doc` 扩展名 | Docling 解析 |
| Markdown | `.md` 扩展名 | 直接读取（仍需理解和标准化） |
| 纯图片 | `.png`/`.jpg`/`.jpeg` | 多模态分析 |
| 混合内容 | 多种类型文件 | 分别处理后合并 |

**1.2 脚本解构**

```bash
uv run twu parse <需求目录>
```

- 调用 Docling 解析 PDF/Word
- 提取文本和图片到 `req/chunks/` 和 `req/assets/`
- 输出图片占位符 `<!-- image: xxx.png -->`

**1.3 Docling 兜底方案**

如果 Docling 解析失败或效果不佳：
- PDF → 降级到 PyPDF2
- Word → 降级到 python-docx
- 记录降级原因，标记 `[解析质量待确认]`

#### Phase 2: 图片理解（多模态分析）

**2.1 Asset 提取（严格）**

确保所有图片都被正确提取：
- 检查 `req/assets/` 目录
- 对比原文档中的图片引用
- 缺失的图片标记 `[图片缺失: xxx.png]`

**2.2 图片分类（思维链）**

对每张图片执行分类推理：

```
Step 1: 观察图片整体特征
- 是否有 UI 元素（按钮、输入框、菜单）？
- 是否有流程箭头或连接线？
- 是否有表格结构？
- 是否有架构层级？

Step 2: 判断图片类型
- 原型图/界面截图：有 UI 元素
- 流程图：有步骤和箭头
- 表格截图：有行列结构
- 架构图：有组件和层级
- 说明图/示意图：概念性内容

Step 3: 记录分类结果
<!-- image-type: 原型图 -->
```

**2.3 针对性分析**

| 图片类型 | 分析重点 | 输出格式 |
|---------|---------|---------|
| 原型图/界面截图 | UI 元素、交互流程、所有可见文字（按钮、placeholder、提示） | 界面描述 + 元素列表 |
| 流程图 | 步骤、分支、条件、连接关系 | 流程步骤列表 |
| 表格截图 | 行列结构、数据内容 | Markdown 表格 |
| 架构图 | 组件、层级、依赖关系 | 架构描述 + 组件列表 |
| 说明图/示意图 | 概念、关系、标注 | 文字描述 |

**2.4 通过图片名称辅助理解**

图片名称可能包含有用信息：
- `login_page.png` → 登录页面
- `order_flow.png` → 订单流程
- `table_user_list.png` → 用户列表表格

#### Phase 3: 阅读理解与标准化

**核心**：还原文档本意，进行必要整理。

1. **阅读所有 chunks**：理解文档整体结构和内容
2. **整合图片描述**：将图片分析结果放回对应位置
3. **标准化输出**：
   - 统一标题层级
   - 整理段落结构
   - 补充缺失的上下文
   - 标记不确定内容 `[待确认]`
4. **生成 index.md**：合并为完整文档，添加来源标注

#### Phase 4: 提交与引导

1. **写入文件**：将 index.md 写入 `req/index.md`
2. **Git Commit**：`req-parse: 解析需求文档`
3. **引导用户**：

```
需求文档已解析：req/index.md

请审核以下内容：
1. 文档结构是否完整
2. 图片描述是否准确
3. [待确认] 的点请确认或修改

审核方式：
- 运行 git diff req/ 查看改动
- 直接修改 index.md 文件
- 或告诉我需要修正的内容
```

#### Phase 5: 审核闭环（多轮对话）

**5.1 用户审核**

用户通过 diff 查看改动：
```bash
git diff req/
```

用户可以：
- 直接修改 req/index.md 文件
- 口头告知 Agent 需要修改的内容
- 发现问题就继续反馈、继续改

**5.2 应用修正**

根据用户反馈修正文档：
- 分析用户修改内容，理解意图
- 应用修正到 index.md
- 可能需要多轮对话

**5.3 完成检查**

1. 检查是否还有 `[待确认]` 标记
   - 有 → 提示用户继续确认
   - 无 → 解析完成
2. 用户确认审核通过
3. 提交更新：`req-parse: 更新需求文档`
4. 引导进入 req-review

### 产物结构

```
req/
├── index.md      # 最终产物（标准化需求文档）
├── chunks/       # 中间产物（按原文件名分片）
└── assets/       # 图片资源
```

### 检查清单

**解构检查**：
- [ ] raw/ 中所有文件都已处理
- [ ] Docling 解析成功（或已降级处理）
- [ ] 所有图片已提取到 assets/

**图片分析检查**：
- [ ] 所有图片已分类
- [ ] 所有 `<!-- image: -->` 占位符已替换
- [ ] 原型图包含所有可见文字、按钮、输入框
- [ ] 流程图包含步骤、分支、条件
- [ ] 表格结构完整（无错行错列）

**标准化检查**：
- [ ] 文档结构清晰
- [ ] 还原了文档本意
- [ ] 无乱码（无法推理的标注 `[待确认]`）
- [ ] 每个来源有 `<!-- 来源：xxx.md -->` 标注

## Skill 2: req-review（测试与需求完备性）

### 目标

从测试视角识别需求问题，将产品答案整合到原文档，审核完备性，最终形成完备的 REQ。

### 输入输出

| 项目 | 内容 |
|------|------|
| 输入 | `req/index.md` |
| 输出 | `req/index.md`（原地修改）+ `req/issues.md` |

### 工作流程

#### Phase 0: 前置检查

1. `req/index.md` 必须存在，否则提示先运行 `/req-parse`
2. 检查 `req/issues.md` 是否存在：
   - 不存在 → 进入 Phase 1（问题识别）
   - 已存在 → 检查是否有未回答的问题，询问用户意图

#### Phase 1: 问题识别

**核心判断标准**：这个问题是否影响测试用例设计或测试结果判断？

**6 个扫描维度**：

| 维度 | 关注点 |
|------|--------|
| 前因后果 | 业务规则、权限约束、前置条件 |
| 边界定义 | 生效条件、范围排除、限制条件 |
| 异常场景 | 失败处理、重试、回退、冲突处理 |
| 交互细节 | 文案、提示、加载、空态、反馈 |
| 数据状态 | 字段定义、状态流转、数据约束 |
| 性能兼容 | 响应时间、并发、兼容范围 |

**过滤规则**：
- ✅ 识别：影响测试用例设计或测试结果判断
- ❌ 过滤：纯业务背景、纯技术实现、常识问题、装饰性问题

#### Phase 2: 生成问题清单

输出 `req/issues.md`

**问题格式**：

```markdown
## 问题 N [分类 · 优先级]

[位置] 第X页 - 章节名称

[问题] 需求文档提到"原文引用"，但没说清楚XXX，导致无法设计测试用例。

[需要澄清]
1. 具体问题1？
2. 具体问题2？

[产品回答]
<!-- 在此填写 -->

---
```

**优先级定义**：
- `critical`：不明确则无法设计测试
- `warning`：影响核心用例设计
- `info`：影响边界和异常用例

#### Phase 3: 提交与引导

1. **写入文件**：将 issues.md 写入 `req/issues.md`
2. **Git Commit**：`req-review: 生成问题清单`
3. **引导用户**：

```
问题清单已生成：req/issues.md

共识别 N 个问题：
- critical: X 个
- warning: Y 个
- info: Z 个

请将 issues.md 发给产品，在 [产品回答] 处填写答案。
填写完成后，告诉我继续合并。
```

#### Phase 4: 答案合并

**前置检查**：
- 检查 critical 问题是否都已回答
- 未回答的 critical 问题 → 提示用户补充

**整合方式**：

1. **直接更新原文**：答案是对原文的补充或修正
2. **添加补充说明章节**：答案是新增的规则或说明
3. **融合到原文**：答案需要与原文深度融合

**操作**：
- 直接修改 `req/index.md`
- Git Commit：`req-review: 合并问题答案`

#### Phase 5: 审核完备性（多轮交互）

**核心**：可能需要多轮交互才能达到最终效果，最终形成完备的 REQ。

**5.1 用户审核**

用户通过 diff 查看改动：
```bash
git diff req/index.md
```

用户可以：
- 直接修改 req/index.md 文件
- 口头告知 Agent 需要修改的内容
- 发现问题就继续反馈、继续改

**5.2 完备性检查**

检查需求文档是否完备：
- 所有 critical 问题已回答并整合
- 业务规则清晰
- 边界条件明确
- 异常处理完整

如果发现新的问题：
- 补充到 issues.md
- 等待产品确认
- 再次合并

**5.3 多轮迭代**

```
发现问题 → 补充 issues.md → 产品确认 → 合并到 index.md → 再次审核
    ↑                                                        ↓
    └────────────────── 继续迭代 ←──────────────────────────┘
```

直到用户确认需求文档完备。

**5.4 完成确认**

1. 用户确认需求文档完备
2. 提交更新：`req-review: 完成需求评审`
3. 引导进入 tc-plan

```
需求文档已完成：req/index.md

需求评审完成，可以运行 /tc-plan 生成测试规划。
```

### 产物结构

```
req/
├── index.md      # 需求文档（req-parse 生成，req-review 修改）
├── issues.md     # 问题清单
└── assets/       # 图片资源
```

### 检查清单

**问题识别**：
- [ ] 所有问题都影响测试用例设计或测试结果判断
- [ ] 没有纯业务背景问题
- [ ] 没有纯技术实现问题
- [ ] 分类准确（6 个维度之一）
- [ ] 优先级准确（critical/warning/info）

**答案合并**：
- [ ] 所有 critical 问题已回答
- [ ] 答案已整合到文档中
- [ ] 新增内容清晰可辨识
- [ ] 保持了原文档结构

## 目录结构

TWU 支持多需求目录：

```
TWU/
├── .claude/skills/           # Skills（所有需求共享）
│   ├── req-parse/
│   ├── req-review/
│   ├── tc-plan/
│   └── tc-gen/
├── twu/cli/                  # CLI 工具
├── CLAUDE.md                 # 业务背景（必填）
├── 需求1/
│   ├── raw/                  # 原始文档
│   ├── req/                  # 需求文档
│   │   ├── index.md
│   │   ├── issues.md
│   │   └── assets/
│   └── tc/                   # 测试用例
│       ├── plan.md
│       └── {Module}/{Scenario}.md
└── 需求2/
    └── ...
```

## 命令速查

| 命令 | 输入 | 输出 |
|------|------|------|
| `/req-parse` | raw/ | req/index.md |
| `/req-review` | req/index.md | req/index.md + req/issues.md |
| `/tc-plan` | req/index.md | tc/plan.md |
| `/tc-gen` | tc/plan.md | tc/{Module}/{Scenario}.md |
| `twu export <dir>` | tc/ | Excel |

## 异常处理

| 场景 | 处理方式 |
|------|----------|
| PDF 加密 | 提示用户提供密码 |
| 文件损坏 | 跳过并记录 |
| Docling 失败 | 降级 PyPDF2/python-docx |
| critical 问题未回答 | 阻止执行答案合并 |
